## 一、飞书开放平台能力边界（机器人视角）

### 1.1 会议内容获取 —— ⚠️ 核心认知：文字内容在 VC，不在 minutes/v1

| 想要的东西 | 正确来源 | CLI 命令 | 权限 scope |
|---|---|---|---|
| 会议 **AI 总结 / 待办 / 逐字稿** | **VC 会议纪要 notes** | `vc +notes --minute-tokens` / `--meeting-ids` | `vc:note:read` |
| 搜已结束会议 | VC | `vc +search`（须≥1个过滤条件） | `vc:meeting.search:read` |
| 录制对应的 minute_token | VC | `vc +recording` | `vc:record:readonly` |
| 妙记**元信息**（标题/时长/url） | minutes/v1 | `minutes +search` | `minutes:minutes.basic:read` |
| 妙记**音视频媒体** | minutes/v1 | `minutes +download` | `minutes:minutes.media:export` |
| 上传音视频**生成**妙记 | minutes/v1 | `minutes +upload` | `minutes:minutes` |

> **坑**：`minutes/v1` **没有 transcript（文字稿）端点**——官方 `lark-minutes` 技能明确写"逐字稿/总结/待办/章节不由本技能处理，请用 `lark-vc`"。很多人会先踩这个坑。

- **触发信号**：事件 `minutes.minute.generated_v1`（妙记生成完成，会议结束 **~10 秒后**触发）。比"会议结束"事件更准——此时文字内容已就绪。
- **套餐前提**：妙记的 **AI 智能总结**属飞书会员/企业版能力，免费版可能只有转写、无 AI 总结或受时长额度限制。**API 能调≠一定有 AI 总结内容**，取决于租户套餐。

### 1.2 群聊消息获取 —— 🔴 全量读取是敏感权限

| 能力 | 端点 / CLI | 权限 | 限制 |
|---|---|---|---|
| 拉群历史消息 | `GET /im/v1/messages` · `im +chat-messages-list` | `im:message.history:readonly` 等 | 机器人**须在群内**；支持 start/end 时间、`page_size` 1–50；`thread` 容器不支持时间过滤；超出套餐存储期的历史拉不到 |
| 实时收 **@机器人** 消息 | 事件 `im.message.receive_v1` | `im:message.group_at_msg`（普通） | 默认只收 @ 它的 |
| 实时收 **群内全量** 消息 | 事件 `im.message.receive_v1` | **`im:message.group_msg`（🔴敏感）** | 需企业管理员审批+应用发布审核，官方"如非必要不建议申请" |
| 搜消息 | `im +messages-search` | user 身份 | 按 chat/sender/时间过滤 |

> **结论**：读"未 @ 机器人的群消息"，无论拉历史还是实时收全量，都属高敏感权限，**需管理员审批**。这是"读群聊更新文档"场景的**最大门槛**。

### 1.3 文档 / 知识库更新

| 能力 | 端点 / CLI | 权限 | 关键限制 |
|---|---|---|---|
| docx 追加/插入块 | `POST /docx/v1/documents/:id/blocks/:block_id/children` · `docs +update` | `docx:document` | 调用身份须是**协作者且有编辑权**；单次≤50块；3次/秒/应用 |
| CLI 高阶更新 | `docs +update --api-version v2` | 同上 | 支持 append / overwrite / str_replace / block_insert_after / block_replace / block_delete / block_move + **Markdown 互转** |
| wiki 节点管理 | `wiki/v2/...` · `wiki +node-*` | `wiki:node:*` / `wiki:space:*` | 节点=容器+一个docx；**更新正文=拿节点 obj_token→走 docx 接口** |

### 1.4 事件订阅：长连接 vs Webhook

| 方式 | 需公网回调 | 加解密验签 | 接入成本 |
|---|---|---|---|
| HTTP Webhook | **需要**（公网域名/内网穿透） | 自己做 | ~1 周 |
| **长连接 WebSocket（SDK / `lark-cli event`）** | **不需要** | **内置，无需自己处理** | ~5 分钟 |

长连接限制：每应用≤50 连接；收到消息须 3 秒内处理；集群非广播（多客户端随机一个收到）。本场景相关事件：`im.message.receive_v1`（群消息）、`vc.meeting.participant_meeting_ended_v1`（会议结束）、`minutes.minute.generated_v1`（妙记生成）。

## 二、开源知识库系统对比（13 个，机器人写入视角）

> 评估核心：**有无官方稳定的程序化写入 API + 能否直接喂 Markdown**。

| 系统 | 协议 | 自托管 | 写入 API | Markdown 原生 | 内置向量检索 | 技术栈 | 部署难度 | Stars | 机器人写入评分 |
|---|---|---|---|---|---|---|---|---|---|
| **Outline** | BSL 1.1 | ✅ | RPC REST，文档完善 | ✅ | ❌ | TS+PG+Redis | 中 | 38.8k | **高** 🥇 |
| **BookStack** | MIT | ✅ | REST，文档完善 | ✅ | ❌ | PHP+MySQL | **低** | 18.8k | **高** 🥈 |
| **SiYuan 思源** | AGPL-3.0 | ✅ | HTTP API，文档完善 | ✅(GFM) | ❌ | Go+TS+SQLite | 中 | 44.3k | **高** 🥉 |
| **MediaWiki** | GPL-2.0 | ✅ | Action/REST，bot 一等公民 | ❌(wikitext) | ❌ | PHP+MySQL | 中-高 | — | 中-高 |
| **Wiki.js** | AGPL-3.0 | ✅ | GraphQL，scope 繁琐 | ✅ | ❌ | Vue+Node+PG | 中 | 28.4k | 中（v3 停滞） |
| **Trilium(Next)** | AGPL-3.0 | ✅ | ETAPI(REST) | 部分(HTML为主) | ❌ | TS+SQLite | 中 | 30k+ | 中（仓库迁移中） |
| **Docmost** | AGPL/企业 | ✅ | 社区版**无** API / 企业版有 | 导入导出 | ❌ | TS+PG+Redis | 中 | 20.5k | 中（企业版↑） |
| **Obsidian** | 闭源(免费) | 数据可同步 | 第三方插件(依赖客户端) | ✅(纯.md) | ❌ | Electron | 高(做后端) | — | 中-低 |
| **Logseq** | AGPL-3.0 | 部分 | 本地 HTTP(依赖客户端) | 块/MD | ❌ | Clojure+Electron | 高(做后端) | 43.3k | 低-中 |
| **AFFiNE** | 混合 | ✅ | GraphQL **无官方文档/不稳定** | 导入导出 | ❌ | TS | 高 | 69.2k | **低** |
| **AppFlowy** | AGPL/开放核心 | ✅ | **无**官方自动化 API | 块/MD | AI内部 | Dart+Rust | 中-高 | 72.0k | **低** |
| Notion(闭源对照) | 闭源SaaS | ❌ | REST 成熟，block 结构+限速 | ❌(转block) | 内部 | SaaS | — | — | 中 |
| Confluence(闭源对照) | 闭源 | DC可 | REST v2 成熟 | ❌(storage格式) | 内部 | SaaS/DC | 高/贵 | — | 中 |

### 推荐三甲的写入 API 示例

**Outline**（首选）：
```bash
curl -X POST https://wiki.company.com/api/documents.create \
  -H "Authorization: Bearer ol_api_xxx" -H "Content-Type: application/json" \
  -d '{"title":"部署手册","text":"# 部署手册\n\n## 步骤一\n...","collectionId":"<uuid>","publish":true}'
# 改文档 POST /api/documents.update {id,text}；建目录 POST /api/collections.create；搜索 POST /api/documents.search
```

**BookStack**（MIT 最干净）：
```bash
curl -X POST https://docs.company.com/api/pages \
  -H "Authorization: Token <id>:<secret>" -H "Content-Type: application/json" \
  -d '{"book_id":12,"name":"API 网关配置","markdown":"# API 网关\n\n配置说明..."}'
# 改页面 PUT /api/pages/{id}；建 book POST /api/books；建章节 POST /api/chapters
```

**SiYuan 思源**（中文/隐私）：
```bash
curl -X POST http://127.0.0.1:6806/api/filetree/createDocWithMd \
  -H "Authorization: Token <token>" -H "Content-Type: application/json" \
  -d '{"notebook":"<notebook-id>","path":"/团队/部署手册","markdown":"# 部署手册\n\n内容..."}'
# 返回新文档ID；块级 /api/block/insertBlock、/api/block/updateBlock
```

> **共性提醒**：① 13 个自托管系统**无一内置开放向量检索**，RAG 需外置（pgvector/Qdrant + 自建 embedding）。② 高频写入要做 **upsert**（机器人侧维护"外部ID→文档ID"映射），否则重复创建。③ 避坑：Trilium 别用已归档的 `TriliumNext/Notes`（指向 `TriliumNext/Trilium`）；AFFiNE/AppFlowy 别因 star 多就选做后端。

## 三、自动化编排平台对比

| 平台 | 协议 | Star | 飞书节点 | LLM 总结 | 触发方式 | 适配本场景 |
|---|---|---|---|---|---|---|
| **n8n** | SUL（自托管内部免费） | ~162k | 社区节点（质量高） | 原生 AI/LangChain | Webhook/Cron/事件/WS | **最佳** |
| **Activepieces** | **MIT** | ~20k | 无原生，靠 MCP/HTTP | 任意 LLM + AI Agent | Webhook/Cron/轮询 | 良好（飞书弱） |
| **Dify(workflow)** | Apache 2.0+ | ~144k | 无官方飞书插件 | 原生（核心） | 定时/webhook 较弱 | 偏"应用"非"管道" |
| **Flowise** | MIT | ~41k | 无 | 原生 chatflow | 事件触发弱 | 不适合事件管道 |
| **Windmill** | AGPLv3 | ~16k | 无（可写脚本调SDK） | 任意 Py/TS LLM 库 | Cron/Webhook | 全代码团队适合 |

- **n8n** 首选：社区节点 [`n8n-nodes-feishu-lark`](https://github.com/zhgqthomas/n8n-nodes-feishu-lark)（MIT，2026 活跃）提供 Lark Trigger（国内 WebSocket/国际 Webhook）+ Bot + MCP，含 Parse Message、Send-and-Wait（人在环路）。

## 四、RAG / 问答平台对比（"知识大脑"，与存储互补）

| 平台 | 协议 | Star | 飞书 connector | 自动同步 | 自带 RAG+问答 | 公司大脑适配 |
|---|---|---|---|---|---|---|
| **Dify** | Apache 2.0+ | ~144k | 无官方 | 工作流可手搭 | 是 | 强，最全能 |
| **RAGFlow** | Apache 2.0 | ~82k | **无** | 内置 Confluence/Notion/GDrive/S3 | 是（检索质量最佳） | 强 |
| **FastGPT** | 部分开源 | ~24k | 无官方（有第三方桥接） | 多源摄入 | 是 | 中-强（中文成熟） |
| **Onyx**（原 Danswer） | MIT(CE)/EE | ~14k+ | **无**（40+其他connector） | 50+ connector 增量同步 | 是（权限感知） | **企业问答最对口** |
| **AnythingLLM** | MIT | ~61k | 无 | 手动/API 为主 | 是 | 中（部署最轻） |
| **Quivr / Khoj** | Apache 2.0 | 数万 | 无原生 | 弱 | 是 | 个人/小团队 |

> **关键**：RAG 平台**普遍无飞书 connector**，接入飞书都要"官方 SDK/CLI/MCP + 自建胶水"——这是工作量主要来源。**Onyx** 权限感知检索最像"公司大脑"；**RAGFlow** 检索质量最佳。存储（飞书Wiki/Outline）与问答（Onyx/RAGFlow）是**互补**关系。
