# 飞书数字员工（书童）· 用户视角体验审视：不足清单 + 改进方案

> 视角：以**新用户 / 管理员 / 普通成员**的第一视角走一遍用户旅程，做 MECE 不足盘点，并给改进方案。
> 对象：`feishu-collector`（信息收集催办）+ `feishu-kb-maintainer`（知识库自动维护）。
> 性质：审视 + 方案，**先 review 再决定实现**。编写日期：2026-06-21。

---

## 0. 核心结论（根因）

两个 skill 都是**反应式手脚**，且都**默认前置条件已就位**——这是不足的总根因：

- **kb-maintainer 假设知识库结构已存在**：routing 要把内容写进某个 Wiki 节点/小节，但新租户里 Wiki 是空的 → 无处可写 → 监听到会议/群聊也沉淀不进去。
- **collector 假设用户已会"咒语"**：上传文件 + @机器人 + 下达收集指令。新用户不知道这套，@ 一下得到的是 `ignore`。
- **整个产品缺 0→1 层**：没有"我是谁、我能干什么、从哪开始"的入口，也没有把空系统填起来的能力。

→ 用户点出的 onboarding 与冷启动，不是附加功能，而是**补上缺失的一整层**。

---

## 1. 用户与旅程（MECE 基线）

**三类用户**：① 管理员/部署者（建库、授权、拥有知识库）；② 发起人（发起收集、定 OKR、推进项目）；③ 普通成员（被 @ 填信息、读知识库）。

**旅程七阶段**（互斥、合起来穷尽用户的完整生命周期）：
1. 首次接触 / 发现 2. 安装配置 / 授权 3. 冷启动 / 空数据 4. 日常使用 5. 功能可发现性 6. 异常 / 空状态 / 错误 7. 信任 / 可观测 / 撤销 / 退出

---

## 2. MECE 不足盘点（按旅程阶段）

### 阶段 1 · 首次接触 / 发现　🔴 缺口最大
| 现状 | 缺口 |
|---|---|
| 机器人被加进群后**沉默**；只有被正确 @ + 带文件/指令才响应。`on-message` 对非命中消息一律 `ignore`，无任何回应。 | ①没有欢迎语/自我介绍；②用户不知道它叫书童、能干什么；③没有"帮助/你能做什么"指令；④首次 @ 得到沉默 = 第一印象就是"坏了/没用"。 |

### 阶段 2 · 安装配置 / 授权　🔴
| 现状 | 缺口 |
|---|---|
| 授权、bot 入群、开 scope 发版、`setup-base`/`setup-route-base` 建库、建 Wiki 骨架——**全是开发者 CLI 操作**，散落在 SKILL.md 的"上线前置"里。 | ①管理员视角无"配置向导"：不知道要授哪些权、bot 要不要入群、库建没建；②没有"体检"能力告诉你"还差什么";③Wiki 骨架要人工手搓，没有脚本生成。 |

### 阶段 3 · 冷启动 / 空数据　🔴 用户重点
| 现状 | 缺口 |
|---|---|
| kb-maintainer 要往**已存在**的 Wiki 节点写；collector 要**已有**一份文件来收集。新公司：Wiki 空、无历史会议、无文档、无名册。 | ①空知识库**无人搭骨架**→沉淀无落点；②没有"主动访谈"把公司基本信息问出来；③没有"引导上传"把存量资料（名册/制度/项目/旧Wiki/Notion 导出）导进来；④面对空系统用户**无从下手**。 |

### 阶段 4 · 日常使用
| 现状 | 缺口 |
|---|---|
| 收集线闭环完整（解析→确认→收集→清洗→写回→群@催办→收工）；沉淀线被动监听会议/群聊/评论+周报。 | ①会议妙记生成但**没匹配到项目**时——目前要么写不进、要么需人工指定，没有"这场会属于哪个项目？要我建页吗"的引导；②收集发起仍依赖"上传文件"，不能"我口述要收哪些字段、找谁"直接起；③群@催办对**未认领**槽位只发"有同学认领一下"，缺少把名单/认领按钮做成卡片。 |

### 阶段 5 · 功能可发现性
| 现状 | 缺口 |
|---|---|
| 能力写在 SKILL.md 里（给宿主 LLM 看），**用户侧无目录**。 | ①用户不知道有"进度查询/暂停/取消/开放登记/OKR 进展/周报"这些能力；②没有随时可呼出的"书童能做什么"菜单；③新能力上线无触达。 |

### 阶段 6 · 异常 / 空状态 / 错误
| 现状 | 缺口 |
|---|---|
| Phase B 已补了不少健壮性（超时/分页/幂等/覆盖保护/参数守卫）。但这些大多是**对内**不崩，**对用户**的话术还不足。 | ①@ 机器人但意图是闲聊→`ignore` 静默，用户不知发生了啥；②权限/403/缺 scope→有抛错但**给用户的解释**不够人话（"我还没拿到 X 权限，请管理员…"）；③收集任务"全员未交到期"→收工汇报有了，但**空知识库/空任务列表**的查询返回空时无引导；④覆盖保护判定 conflict→"转发起人确认"目前是设计约定，**确认的具体交互**（卡片/话术）未定义。 |

### 阶段 7 · 信任 / 可观测 / 撤销 / 退出
| 现状 | 缺口 |
|---|---|
| 写知识库/Task/OKR 后多为静默；状态在 Base 可审计但用户看不见。 | ①机器人写了什么，用户**当下无感知**（没有"我把 X 写进了 Y〔链接〕"的回执）；②**无撤销**：写错了只能人工去 Wiki 改；③无"本周我帮你做了哪些"的可读小结（周报是知识汇总，不是"机器人行为账单"）；④退出/停用：单个任务能暂停取消，但"让书童整体安静下来/退场"无入口。 |

## 3. 改进方案

### 🌟 重点 A：新增 `feishu-init`（书童引导）编排 skill —— 补"发现 + 配置 + 上手"

一个面向**用户/管理员**的引导编排器（区别于面向宿主 LLM 的 SKILL.md），触发于：bot 入群/首次对话、或用户说"你好/帮助/书童能做什么/开始/init"。

**它做三件事：**
1. **自我介绍 + 能力菜单**（交互卡片）：一句话说清"我是书童，能①主动帮你收集信息并催办、②自动把会议/群聊沉淀进知识库"，配 2-3 个可点的"立即试试"按钮。
2. **配置体检（管理员向）**：逐项检查并报告——user 是否授权齐 scope、bot 是否入群、状态库 Base 建没建、Wiki 知识空间在不在；缺什么就给**人话指引 + 可执行入口**（能自动做的直接做：`setup-base`/`setup-route-base` 建库、生成 Wiki 骨架）。
3. **三条上手路径**（解决"从哪开始"）：
   - 「帮我收集一份信息」→ 进 collector（若没文件，转下面 B 的访谈式/模板）。
   - 「帮我搭起公司知识库」→ 进冷启动方案 B。
   - 「看看你能做什么」→ 能力清单（见重点 C）。

> 复用：体检调 lark-cli 探测；建库复用现有 setup 脚本；卡片用 `feishu-shared/message`。**新增的是"编排 + 话术 + 卡片"，不是新手脚。**

### 🌟 重点 B：冷启动三件套 —— 补"空数据→有数据"

**B1. 自动搭知识库骨架（scaffold，前置）**
新增 `kb-scaffold` 能力：按设计文档的标准目录树，用 `wiki +node-create` 一键生成知识空间 + 标准节点（公司总览 / 项目空间 / 会议纪要库 / 群知识沉淀 / 公司待办看板 / OKR / 制度SOP）。**这是 kb-maintainer 能工作的前提**——有了节点，routing 才有落点。

**B2. 提问 / 访谈式主动收集（seed by interview）**　← 用户重点
内置一套"公司知识库初始化访谈"：书童**主动**逐项发问，把答案写进对应 Wiki 节点。问题模板（可裁剪）：
- 公司一句话是做什么的？主要业务线/团队有哪些？
- 现在在推哪些项目？各自负责人？
- 本季度/年度 OKR 或战略目标是什么？
- 有哪些必须知道的制度 / SOP / 常见问题（FAQ）？
- 团队成员名册（姓名+角色），用于后续 @ 与人名解析。

> **直接复用 collector 的机器**：把"访谈问题"建成槽位（每问一槽），责任人=发起人/指定人，**落点改成 Wiki 节点而非用户文件**。即 `collect-* 原子 + kb-route/doc` 的组合编排，不需重写收集逻辑。这正是原子化重构后能低成本拼出的新场景。

**B3. 引导上传导入（seed by import）**　← 用户重点
书童引导用户："把你现有的资料发我——员工名册、制度文档、项目清单、历史会议纪要、旧 Wiki/Notion 导出都行。" 然后 `file`(解析/本地转在线) → `kb-extract`(归类) → `kb-route`+`doc`(写进骨架对应节点)。对名册类，顺带建立人名↔open_id 映射，喂给后续 `contact` 解析。

> B1 是地基，B2/B3 是两种填充方式（一个靠问、一个靠导），用户可任选或都用。

### 重点 C：功能可发现性
- **「书童能做什么」常驻指令**：任何时候呼出能力菜单（收集/催办/进度/暂停取消/开放登记/知识沉淀/OKR进展/周报/初始化）。
- **空状态即引导**：会议没匹配到项目 → "这场会属于哪个项目？要我建页吗"；@ 了但意图不明 → 给菜单而非沉默。
- **新能力触达**：周报里附一栏"你还没用过的能力"。

### 重点 D / E / F：异常·信任·退出
- **意图兜底**：`on-message` 的 `ignore` 分支对 @机器人 的消息改为"没太懂，你是想 [收集/查进度/初始化]？"小菜单，不静默。
- **错误人话化**：权限/403/缺 scope 统一转成"我还差 X 权限，请管理员到 Y 开通"的用户话术（技术细节进日志）。
- **写入回执 + 撤销**：写知识库/Task/OKR 后回一条"已把〔摘要〕写进〔链接〕，不对的话回复'撤销'"；撤销复用 routing 台账（有 target_locator，可定位回滚）。
- **conflict 确认交互**：覆盖保护判 conflict 时，发卡片给发起人"目标被人改过，要①覆盖②保留③合并"，把"转确认"落成具体按钮。
- **整体退场**：加"让书童安静/暂停所有自动沉淀"的总开关指令。

## 4. 优先级与落地建议（复用原子，增量小）

| 优先级 | 项 | 复用 | 价值 |
|---|---|---|---|
| **P0** | `feishu-init` 引导卡片 + 能力菜单（重点 A1/C） | message 卡片 | 解决"沉默/不知能干啥"，第一印象 |
| **P0** | `kb-scaffold` 知识库骨架生成（B1） | wiki node-create | kb-maintainer 能工作的前提 |
| **P1** | 配置体检向导（A2） | lark-cli 探测 + setup 脚本 | 管理员上手 |
| **P1** | 访谈式冷启动（B2） | collect-* 原子 + kb-route/doc | 空数据→有数据（问） |
| **P1** | 引导上传导入（B3） | file + kb-extract + kb-route | 空数据→有数据（导） |
| **P2** | 写入回执 + 撤销、conflict 卡片、意图兜底、错误人话化（D/E/F） | routing 台账 + message | 信任与容错 |

> 形态选择待定：`feishu-init` 做成**第三个编排器 skill**（与 collector/kb-maintainer 平级），下面挂 `kb-scaffold`、访谈、体检等；冷启动 B2 复用 collector 原子、B3 复用 kb 原子——符合"编排器编排原子"的现有架构。
