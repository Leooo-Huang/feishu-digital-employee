# 飞书数字员工 · 功能与实现方案

## 一、概述

飞书数字员工是一个独立常驻、不绑定任何聊天会话的智能体，围绕公司在飞书内的协作场景提供两大能力：**知识库自动维护**与**信息收集助手**。前者被动沉淀——监听会议、群聊与文档协作，将散落的纪要、决策、待办、目标进展抽取并写入知识库；后者主动收集——接收上传的文档或表格、按指令在群里或私信主动对话，逐项收齐信息后清洗确认、写回源文件、自动催办并收工汇报。

两条能力线共享同一套运行时、同一套能力层、同一套持久化层，构成单一数字员工的两组技能。运行时为 Hermes Agent，提供 GLM 推理大脑（Hermes 配置的 GLM 模型，当前为 glm-5-turbo）、飞书对话入口、常驻服务、终端工具与定时调度；能力层为飞书 CLI（lark-cli），承担一切飞书数据的读写操作；持久化层为飞书多维表格（Base），所有运行状态存于飞书云端。每次处理都是"从 Base 读状态 → 推理决策 → 调 lark-cli 执行 → 写回 Base"的无状态循环，因此跨重启不丢、可在飞书内直接审计。

知识库线以"路由幂等表"统一去重与定位；收集线以"任务表 + 槽位表"承载收集活动的完整生命周期。GLM 大脑承担两类推理：知识库线的要点抽取与目标识别，收集线的计划生成、答案映射与消歧。

## 二、功能

### 2.1 模块一 · 知识库自动维护（被动沉淀）

知识库以飞书 Wiki 知识空间为骨架，按项目组织，全局索引页只放自动生成的链接以保持单一真身、多视图。模块一包含七项功能。

#### 知识库文件结构（Wiki 知识空间骨架）

知识空间的目录树明确每个节点由哪项功能维护，为路由层提供确定的目标形状。

```
公司知识库（飞书 Wiki 知识空间）
├── 公司总览
│    ├── 公司战略与年度目标        ← 功能 4 按周镜像原生 OKR
│    ├── 重要决策记录 Decision Log  ← 功能 1、2 抽取的"拍板"决策自动汇总
│    └── 组织架构与成员            ← 人工维护
├── 项目空间（每项目一子页，主轴）
│    └── 项目X
│        ├── 概述 / 需求 / 设计     ← 人工维护
│        ├── 会议纪要              ← 功能 1 自动追加
│        ├── 群聊沉淀              ← 功能 2 自动更新
│        └── 待办汇总              ← 功能 3 镜像原生 Task（只读）
├── 会议纪要库（按时间索引）        ← 功能 1 自动生成的纪要链接列表
├── 群知识沉淀（按群索引）          ← 功能 2 自动维护
├── 公司待办看板                   ← 真身为原生 Task，本页只读汇总（功能 3）
├── OKR / 战略目标                 ← 真身为原生 OKR，本页可读汇总（功能 4）
└── 制度 / SOP / FAQ              ← 人工维护
```

路由幂等表与原始素材冷存不在 Wiki 内，属于系统的状态存储，定位见第四章。

#### 功能 1 · 会议沉淀

会议结束后，妙记纪要生成（约会后十秒）触发会议线。系统抓取妙记的 AI 总结、决策、待办与逐字稿，按会议主题、关联日历或参会群命中目标 Wiki 节点，将一段以日期标记的会议纪要追加进对应项目文档的「会议纪要」小节，同时把决策汇入「重要决策记录 Decision Log」、把行动项推进待办与战略进展。企业套餐妙记自带 AI 总结时直接采用；无 AI 总结时由 GLM 二次总结。

#### 功能 2 · 群聊沉淀

群有新消息时触发群聊线，或由定时任务按时间窗批量拉取。原始群消息信噪比低，不逐条入库：累积一个时间窗或达到一定消息数后，由 GLM 抽取决策、结论、待办与 FAQ，丢弃闲聊，按群 chat_id 映射到对应文档，更新「群聊沉淀」小节，写入前与既有内容去重；其中"拍板"决策汇入「重要决策记录 Decision Log」。群全量读取在具备敏感权限时实时进行；不具备时退化为"@机器人主动归档"——只沉淀显式 @机器人 的消息。

#### 功能 3 · 待办维护（原生 Task）

从会议与群聊中抽取的行动项写成飞书原生 Task，带负责人、截止时间与提醒，任务内回链来源会议或消息。文档侧的「待办汇总」与「公司待办看板」是原生 Task 的只读镜像，勾选与修改在飞书 Task App 内完成，单一事实源在原生 Task。

#### 功能 4 · 战略目标维护（原生 OKR）

战略目标由人设定，系统不新建 Objective，只写进展：GLM 识别会议与群聊中与某 KR 相关的进展，挂到对应 KR 写成原生 OKR 进展记录。「公司战略与年度目标」页按周镜像成可读汇总，单一事实源在原生 OKR。

#### 功能 5 · 文档评论智能回复

文档协作过程中产生的评论触发文档评论线。系统读取所在文档正文，理解评论上下文后在评论串内回复。该功能的触发事件 EventKey 待探测（按 lark-cli `event list` 实测的文档评论事件名为准），探测前以「@机器人 的文档内提及」为兜底触发源。每条评论的回复以路由幂等表登记一条 source_type=comment 记录，按评论标识去重，避免对同一条评论重复回复。

#### 功能 6 · 智能路由 + 幂等

路由层决定每条外部源写到哪个文档、任务清单或 OKR，并对每次写入做去重。每条源按来源标识查询路由幂等表：未记录则写入并记录落点与内容指纹；已记录且内容指纹变化则按落点覆盖更新；指纹未变则跳过。路由幂等表是各条沉淀线的写入前置闸，杜绝重复写与写错文档。

#### 功能 7 · 周期汇总

定时任务每周把散落的决策、待办与 OKR 进展自动汇总成「周报」页，追加进知识库对应汇总文档。

### 2.2 模块二 · 信息收集助手（主动收集）

发起人上传一个含待收集信息的文档或表格、@机器人下达指令，系统据此在群里或私信主动对话收集，逐项收齐后清洗、确认、写回源文件，并自动催办、收工汇报。

#### 三种收集形态

- **表格按行收集**：每人填自己所在行。解析源文件时按"行 × 可填列"预建槽位，责任人来自行内人名列，收齐后原地填回对应单元格。
- **问题清单 / 向特定人**：面向指定责任人逐项提问。解析时按"每问一槽"预建槽位，责任人来自文件责任人列或指令 @点名，收齐后填回对应位置。
- **开放式登记**：面向不特定人开放上报。槽位在每份上报提交时动态生成，责任人为提交者本人，收齐后追加新行。

三种形态统一为同一"槽位"模型，差异仅在三处：槽位预建还是提交时建、责任人预绑还是提交时捕获、原地填格还是追加新行；其余逻辑共用。

#### 四种责任人来源

- 文件责任人列：从源文件中的人名列解析。
- 指令 @点名：从发起指令中的 @提及解析。
- 全群谁领谁填：面向全群开放，谁应答谁认领对应槽位。
- 通讯录智能匹配：按通讯录与群成员对人名做智能匹配。

四种来源统一经由 lark-contact 做人名解析，输出唯一、多候选或零匹配三种结果。解析支持按群成员范围（chat_id）优先匹配，对重名场景提高消歧准确率。零匹配或多候选无法确定时，槽位标待澄清，并在发起确认环节把无法匹配的人名列给发起人确认，或在群里 @所有人认领。

#### 三种场所

收集场所由任务发起时选定：群内公开 / 私信逐人 / 群与私信混合。混合场所的语义是"群公告 + 私信收集"——发起确认与完成汇报走群、公开可见，逐项发问按责任人走私信，与单一场所的区别在收集环节的可见性。

#### 两道确认闸

- **第一道（写真人前 · 发起确认）**：解析出收集计划后，先在群里发交互卡片（含确认与取消按钮），把"收集哪些字段、问哪些人、什么场所、什么截止时间、什么催办策略"复述给发起人，得到确认后才开始向真人发问。
- **第二道（写回前 · 复述确认）**：把清洗后的值复述给提供者确认无误，再写回源文件。

#### 数字员工行为

自动追问与清洗不合格回答；带截止的定期催办，临近截止升级飞书加急；进度可随时查询；全部收齐或到期后自动发完成汇报并附填好的文件链接。缺失值绝不编造，到期未交如实列入未交名单。截止后迟到回答的默认策略为收下并补填、标注"迟交"，可按任务级配置改为拒收。

## 三、总体架构

数字员工分三层：Hermes Agent 运行时、lark-cli 能力层、飞书 Base 持久化层。两大功能模块共用这三层。

### 3.1 Hermes Agent 运行时（驱动）

Hermes Agent 是数字员工的运行时与驱动，提供以下能力：

- **GLM 大脑**：以 Hermes 配置的 GLM 模型（当前为 glm-5-turbo）承担全部推理——知识库线的要点抽取、决策识别、目标进展识别、文档评论上下文理解；收集线的收集计划生成、答案到槽位的映射、人名消歧、清洗判定与意图分类。
- **飞书对话入口**：通过 IM channel 收发文本与文件，并对文档评论做智能回复。会议、群聊、收集对象的回复、发起人的指令与进度查询，均经此入口进出。
- **常驻服务**：以 systemd user 服务 hermes-gateway 形式常驻，独立于任何聊天会话运行。
- **terminal 工具**：以 shell 调用 lark-cli，承接大脑对飞书数据的一切读写动作。
- **cron 调度**：定时触发心跳——知识库线的定时拉取与周期汇总，收集线的催办 tick。
- **skill 挂载**：技能挂载于 `~/.hermes/skills/` 目录，大脑读取技能指令执行工作流。

### 3.2 lark-cli 能力层（手脚）

lark-cli 承担一切飞书数据操作，是数字员工的手脚。业务子命令带 `+` 前缀。base、sheets、contact、im、docs、drive 六个域的命令名已按实测命令面锁定；task、okr、vc、minutes、wiki、event 六个域的命令名待探测，以实测命令面为准。能力覆盖多维表格读写、在线表格单元格读写、通讯录人名解析、即时消息收发与加急、文档读写、云空间文件导入与解包、事件订阅、会议纪要、原生 Task、原生 OKR、知识库节点。

通讯录人名解析统一以 `contact +search-user --query '<名字>' --as user` 调用：search-user 必须以 user 身份执行，返回 users[]，每项含 open_id 与 name；可加 --user-ids 按 open_id 精确查、--queries 多关键词并行。

### 3.3 飞书 Base 持久化层（记忆）

全部运行状态存于飞书多维表格，禁止 SQLite 或任何外部数据库。知识库线用一张「路由幂等表」承载源到目标的映射与去重；收集线用「任务表 + 槽位表」承载收集活动的完整状态。状态存于飞书云端的好处是重启不丢、可在飞书内直接审计，并使运行时层保持无状态可移植。

```
会议/群聊/文档评论 ──┐                              发起群/私信 ──┐
群成员回复/指令 ──────┼─消息/事件─▶ [Hermes IM channel]          │
                     │                    │                     │
调度器(cron) ─tick────┼────────────────────┤                     │
                     │              [GLM 读 skill 决策]          │
                     │                    │                     │
                     └─────────▶ [terminal 调 lark-cli] ◀────────┘
                                          │
                                  [飞书 Base 持久化]
                          路由幂等表 / 任务表+槽位表
```

## 四、数据持久化模型

### 4.1 路由幂等表（知识库线）

| 字段 | 含义 |
|---|---|
| source_type | meeting / chat / comment / manual |
| source_id | 妙记 minute_token / 消息 message_id / 群+时间窗 / 评论 comment_id |
| source_meta | 会议主题 / 群名 / 文档标题 / 时间 |
| target_kind | doc / task / okr / comment |
| target_id | docx document_id / wiki node_token / task_guid / okr KR 标识 |
| target_locator | 落点：doc→block_id；sheet→SheetId!单元格；base→record_id+字段；comment→评论串标识 |
| content_hash | 内容指纹，判断是否需要更新 |
| status | written / updated / skipped |
| last_synced_at | 最近同步时间 |

### 4.2 任务表（收集线，一条 = 一次收集活动）

| 字段 | 含义 |
|---|---|
| task_id | 主键 |
| 标题 | 收集活动名称 |
| 状态 | 草稿 / 待确认 / 收集中 / 暂停 / 已完成 / 已取消 |
| 场所 | 群 / 私信 / 混合 |
| 发起群chat_id | 来源群 |
| 发起人 | open_id |
| 源文件 | token + 类型（sheet / base / doc / 本地转换） |
| 目标文件 | 默认等于源文件 |
| 截止时间 | deadline |
| 催办策略 | 间隔小时 + maxAttempts，临近截止升级加急 |
| 原始指令 | 发起人 @机器人 的指令原文，作为 GLM 推理输入 |
| 收集计划摘要 | GLM 生成的自然语言收集计划 |
| created_at / updated_at / completed_at | 时间戳 |

### 4.3 槽位表（收集线，一条 = 一个待收集信息点）

| 字段 | 含义 |
|---|---|
| slot_id | 主键 |
| task_id | 关联任务 |
| 字段名 | 待收集字段，如身份证号 / T恤尺码 / 预算 |
| 对象 | 关于谁或哪一行；开放式登记下为提交编号 |
| 责任人open_id | 解析后的飞书用户，开放式可空 |
| 责任人原文 | 解析前文本 |
| 落点 | 精确写回位置：sheet→单元格 / base→record_id+字段 / doc→block 锚点 |
| 值 | 清洗后的值 |
| 状态 | 待问 / 已问 / 收到原始 / 清洗中 / 待确认 / 已填 / 跳过 / 不适用 / 待澄清 |
| 内容指纹 | content_hash，幂等用 |
| 追问次数 | attempts |
| 最近询问时间 | last_asked_at |
| 来源 | 消息标识或链接，留痕 |

## 五、数据流与触发

### 5.1 知识库线

会议沉淀：会议结束 → `event consume minutes.minute.generated_v1`（事件名待探测）触发 → `vc +notes` 取妙记纪要（妙记 AI 总结或外接 GLM 总结）→ 查路由幂等表定位目标 → `docs +update` 追加纪要 → `task +create` 写待办、`okr +progress-create` 写进展。

群聊沉淀：群有新消息 → `event consume im.message.receive_v1`（事件名待探测），或定时 `im +chat-messages-list` 按时间窗拉取 → GLM 抽取决策/结论/待办/FAQ → 查路由幂等表 → `docs +update` 更新群聊沉淀小节，拍板决策汇入 Decision Log。

文档协作：文档有新评论 → 文档评论事件（EventKey 待探测，探测前以 @机器人 文档内提及为兜底）→ `docs +fetch` 读文档正文 → GLM 理解评论上下文 → 在评论串内回复 → 路由幂等表登记评论标识，避免重复回复。

周期汇总：cron 定时 → 汇集决策/待办/OKR 进展 → `docs +update` 写周报页。

### 5.2 收集线

发起：发起人在群里 @机器人 并附文件，或收集对象在私信回复 → Hermes 飞书 channel 接入 → GLM 读 SKILL.md 决策 → terminal 调 lark-cli：`drive`/`sheets`/`base` 解析源文件、`contact` 解析人名、`im` 发问与回复、`sheets`/`base`/`docs` 写回、`base` 记任务与槽位状态。

消息归属判定：私信来的人按"该人有待填槽位的活跃任务"路由；当此人同时在多个收集任务里、回复无法唯一归属时，由 GLM 反问消歧（"你说的是【团建报名】还是【报销信息】"）。群里 @机器人 的消息先由 GLM 做意图四分类：新发起指令 / 进度查询 / 开放式登记上报 / 暂停取消指令。

催办：cron 触发 tick → lark-cli `im` 催办，临近截止 `im messages urgent_*` 加急。

收集线只处理 @机器人 的发起与进度消息、以及收集对象的定向回复（私信或 @），天然不依赖群全量消息的敏感权限。

## 六、各功能实现路径

### 6.1 会议沉淀实现路径

`event consume minutes.minute.generated_v1`（事件名待探测）监听会议结束 → `vc +notes` 取 summary/todos/逐字稿（`vc +search`/`vc +recording` 辅助定位会议与 minute_token，`minutes +download` 按需取音视频）→ GLM 在妙记无 AI 总结时二次提炼 → `base +record-search` 查路由幂等表 → 命中目标文档后 `docs +update --api-version v2` 追加纪要段；行动项 `task +create`、目标进展 `okr +progress-create`；写入结果 `base +record-upsert` 回写路由幂等表。Wiki 节点正文经 `wiki +node-get`（命令名待探测）取 obj_token 后由 `docs +update` 写入。task、okr、vc、minutes、wiki、event 域命令名以实测命令面为准。

### 6.2 群聊沉淀实现路径

`event consume im.message.receive_v1`（事件名待探测）或定时 `im +chat-messages-list` 按时间窗拉群消息（`im +chat-search` 按群名取 chat_id）→ 累积时间窗后 GLM 抽取有沉淀价值要点 → `base +record-search` 查路由幂等表去重 → `docs +update` 更新群聊沉淀小节 → `base +record-upsert` 记录指纹与落点。

### 6.3 待办维护实现路径

行动项经 GLM 抽取后 `task +create` 建原生 Task（负责人/截止/提醒），来源回链写入任务；后续变更用 `task +patch`。文档侧待办看板的只读镜像由 lark-cli 读原生 Task 渲染——按 `task +get-my-tasks` 或 `task tasklist tasks` 取任务后写入「公司待办看板」与项目页「待办汇总」。task 域写入与读取命令名均以实测命令面为准。

### 6.4 战略目标维护实现路径

GLM 识别与 KR 相关的进展 → `okr +progress-create` 挂到对应 KR。「公司战略与年度目标」页按周读 OKR 镜像为可读汇总——按 `okr +cycle-detail` 读周期数据渲染汇总。okr 域命令名以实测命令面为准。

### 6.5 智能路由 + 幂等实现路径

每条源以 `base +record-search` 按 source_id 查路由幂等表；未命中则写入目标并 `base +record-upsert` 记录 target_locator 与 content_hash；命中且 content_hash 变化则按 target_locator 覆盖更新（doc 按 block_id、sheet 按 SheetId!单元格、base 按 record_id+字段）；指纹未变则跳过。

### 6.6 周期汇总实现路径

cron 定时遍历周期内的决策/待办/OKR 进展 → GLM 汇编 → `docs +update` 追加周报页。

### 6.7 收集发起实现路径

文件引用解析：飞书链接经 `drive +inspect` 拿 type/title/canonical token（含 wiki 解包）；本地文件经 `drive +import --file <local> --type docx|sheet|bitable|slides` 转在线，`drive +task_result` 轮询取新 token（转换失败报错，不静默 fallback）。结构读取：`sheets +cells-get --range '<SheetId>!A1:Z' --as user` 读在线表格、`base +field-list`/`base +record-search` 读多维表格、`docs +fetch` 读文档。GLM 据结构与指令生成收集计划：判定形态、定字段集与每槽的对象与落点、定场所与截止与催办策略。责任人原文逐个经 `contact +search-user --query '<名字>' --as user --json` 解析（唯一直接用，多候选交发起人消歧，零匹配标待澄清；可按群成员范围优先匹配以消歧重名）。第一道确认闸：在群里发交互卡片（含确认与取消按钮，正文为计划摘要）发起确认。确认后 `base +record-upsert` 落任务表与槽位表，任务置收集中；首轮询问按场所 `im +messages-send --user-id ou_xxx --as bot --idempotency-key <slot_id+轮次>`（私信）或 `--chat-id oc_xxx --as bot`（群），槽位转已问。

### 6.8 收集推进实现路径

收集对象回复 → Hermes channel 接入 → `base +record-search` 载入此人未终结槽位 → GLM 将回复映射到槽位 → 清洗归一化（身份证/尺码/邮箱/日期按内置规则，其余按字段语义校验），不合格则 `im +messages-send` 追问并留清洗中 → 合格后第二道确认闸复述确认 → 写回：`sheets +cells-set --range '<SheetId>!C5' --cells '[[{"value":"L"}]]' --as user`（表格）、`base +record-upsert --base-token <t> --table-id <tbl> --record-id <rec> --json '<fieldMap>' --as user`（多维表格）、`docs +update --api-version v2`（文档），按内容指纹比对决定写或跳过；成功后 `base +record-upsert` 更新槽位为已填并回写指纹。开放式登记追加新行；定向收集改答覆盖原值并更新指纹留痕。

### 6.9 进度查询实现路径

`base +record-search` 聚合该任务槽位状态 → `im +messages-send` 回"已收 X/Y，还差〔责任人 + 字段〕"。

### 6.10 催办与收工实现路径

cron 触发 tick → `base +record-search` 遍历状态为收集中的任务 → 按催办间隔与 maxAttempts 挑选未填槽位 → `im +messages-send --idempotency-key <slot_id+轮次>` 发提醒、追问次数加一、更新最近询问时间，记录该提醒返回的 message_id。临近截止对私信加急：`im messages urgent_app --message-id <om_xxx>`（或 urgent_phone / urgent_sms），其中 message_id 取自该提醒消息 messages-send 的返回值——加急作用于"先发出的那条提醒消息"，要求 bot 为该消息发送者且在会话内。全部已填或到期 → `im +messages-send` 群里发完成汇报（含未交名单）并附文件链接 → 任务置已完成。

## 七、关键机制

### 7.1 幂等

幂等分写回侧与发送侧两层。

写回侧以 content_hash 为准做 upsert：内容指纹与既有相同则跳过，不同则按落点覆盖更新。知识库线在路由幂等表记录每条源的 target_locator 与 content_hash；收集线在槽位表记录每个槽位的内容指纹。每次处理从持久化层重读状态，重启或重跑绝不重复写或填错位置。

发送侧以 idempotency-key 为准防重复发送：调 `im +messages-send` 时带 `--idempotency-key`（如 slot_id 与轮次的组合），重启重放同一条消息不重复发问、不重复催办，与写回侧 content_hash 幂等互补，共同覆盖重放与并发一致性。

### 7.2 收集槽位状态机

主路径：待问 → 已问 → 收到原始 → 清洗中 ⇄ 追问 → 待确认 ⇄ 改正 → 已填。旁支：待澄清（人名无法解析，解析后回待问）、不适用、跳过（到期未交）。已填后改答经 reopen 回清洗中；跳过的槽位在截止后有迟交回答时经 reopen 回待问补收。任务级状态机：草稿 → 待确认 → 收集中 →（暂停）→ 已完成 / 已取消。每条回复独立推进各自槽位，靠 per-slot 状态加幂等防竞态。

### 7.3 清洗归一化

身份证：全角转半角、去空格、字母大写为 X，校验 18 位（17 位数字 + 末位数字或 X）。尺码：全角半角与大小写归一后，校验属于允许集 XS / S / M / L / XL / XXL / XXXL。邮箱：全角半角归一后做格式校验。日期：归一到 YYYY-MM-DD（兼容斜杠、中文年月日等分隔）。非内置类型按字段语义校验。不合格、缺失或歧义则追问，停在清洗中。

### 7.4 催办

按催办间隔与 maxAttempts 从未填槽位中挑选需提醒者，每次提醒经 `im +messages-send` 发出并记录返回的 message_id；临近截止（阈值内）将提醒升级为飞书加急，加急作用于该条提醒消息，带 `--message-id <om_xxx>`。加急为 bot-only 且要求 bot 是该消息发送者并在会话内，故默认先普通提醒、临近截止再升级，每人加急次数有上限。

### 7.5 两道确认闸

写真人前发起确认（向发起人发交互卡片复述收集计划，卡片含确认与取消按钮）、写回前复述确认（向提供者复述将写入的值）。前者防错向真人发问，后者防错值写回文件。

### 7.6 写回保护

写文档或表格需机器人是协作者且有编辑权。捕获 403 时不静默失败，提示发起人把机器人加为该文件协作者。目标位置若已被人工修改且与将写入不一致，不覆盖，转发起人确认。

### 7.7 不静默 fallback 与不编造

本地文件转在线失败必须报明原因，禁止静默退化。缺失值绝不编造，到期未交如实列入未交名单。

## 八、部署

### 8.1 Hermes 挂载与常驻

技能挂载于 `~/.hermes/skills/`，GLM 大脑读取技能指令执行工作流。运行时以 systemd user 服务 hermes-gateway 常驻，独立于任何聊天会话；飞书对话经 IM channel 进出，文档评论经文档评论智能回复进出。每次处理无状态——读 Base、推理、调 lark-cli、写回 Base。

### 8.2 cron 催办与定时

cron 承担两类定时心跳：收集线的催办 tick（遍历收集中任务、挑选并发送提醒、临近截止加急、收工汇报）；知识库线的群聊定时拉取与周期汇总。

### 8.3 权限 scope

scope 标识符与飞书后台命名对齐，分"已具备"与"上线前必补 / 待确认"两栏。

已具备（user 身份已授权）：

- base:app:update —— 路由幂等表、任务表、槽位表读写
- contact:user.base:readonly、contact:user.basic_profile:readonly —— 人名解析
- docx:document:create、docx:document:write_only、docx:document:readonly —— 文档写回与读取
- docs:document:import —— 本地文件（Excel/Word/PPTX/Markdown）导入转在线
- drive:file:upload、drive:file:download —— 本地文件上传与文件下载
- im:message:readonly、im:message.p2p_msg:get_as_user —— 消息读取与单聊取消息
- wiki:node:create、wiki:space:read —— 知识库节点创建与空间读取
- offline_access —— 令牌自动刷新

上线前必补 / 待确认（阻塞发送与写回）：

- im:message:send_as_bot —— bot 发送 DM / 群消息与加急。当前只见 im:message:readonly，发送 scope 须在开发者后台开通并发版（阻塞催办与提问发送）
- sheets / spreadsheet 写 scope —— 在线表格写回所需。user 身份当前未见显式表格写 scope，写在线表格前确认（阻塞表格写回）
- bot 入群 —— 加急（urgent_*）与 bot 发群消息要求 bot 是群成员（阻塞群发与加急）

以下 scope 由系统功能要求，按对应飞书后台标识开通：vc:note:read（妙记纪要读取）、task:task:write 与 task:task:read、task:tasklist:read（原生 Task 写入与镜像读取）、okr:okr.content:writeonly 与 okr:okr.period:readonly（原生 OKR 进展写入与按周镜像读取）。

知识库群全量读取需敏感权限 im:message.group_msg（管理员审批），否则退化为"@机器人主动归档"。收集线只处理 @机器人 与定向回复，不依赖该敏感权限。

### 8.4 bot 入群

bot 须加入目标群：飞书加急与群消息发送均要求 bot 是群成员，加急还要求 bot 为消息发送者且在会话内。bot 发送消息权限在开发者后台开通并发版。
