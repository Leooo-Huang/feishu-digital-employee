# 飞书数字员工 · 原子化重构设计（拆分 + 编排）

> 目标：把 `feishu-collector`、`feishu-kb-maintainer` 两个"聚合大功能"的 skill，拆成**可单独调用、可复用的原子 skill**，大 skill 改造成**只负责编排调度的编排层**（范式参照 autodev：总编排器编排多个子 skill）。
> 约束：拆分后**功能等价、对外行为不变**（batch1+2 的 bug 修复是**有意的行为修正**，在新结构上一次性落地，单列说明）。
> 编写日期：2026-06-09。

---

## 一、现状分析：两个大 skill = 小功能聚合

| 大 skill | 现有模块（导出函数） | 内含的可独立能力 |
|---|---|---|
| feishu-collector | larkcli, hash, base-io, file-parse, contact, clean, state, schedule, messaging, locator + bin(on-message/tick/setup-base) | 文件解析、收集计划、人名解析、清洗、槽位状态机、催办挑选、消息收发/群@/加急、落点写回、状态持久化、收工(缺) |
| feishu-kb-maintainer | larkcli, hash, route-io, minutes, extract, kb-write, task-io, okr-io, comment + bin(on-event/digest/setup-route-base) | 妙记纪要、群聊拉取、抽取归类、知识库写、原生Task、原生OKR、评论回复、路由幂等、周期周报 |

**问题**：① `larkcli.js`、`hash.js` 两份重复；② 每个大 skill 的 SKILL.md 既当大脑又当文档，能力无法被别处复用；③ 编排逻辑与能力逻辑揉在一起。

**重构收益**：原子能力可被任意编排器/未来新 skill 复用；共享底座只修一次；编排层薄、易读、易测；契合 autodev"编排器 + 子 skill"范式。

---

## 二、目标架构（三层 + 编排器）

```
┌─ 编排层（大 skill 瘦身为编排器）────────────────────────────┐
│  feishu-collector(编排)      feishu-kb-maintainer(编排)      │
│   on-message 路由+意图分类     on-event 路由 + cron digest    │
│   按 SKILL 工作流串原子        按 SKILL 工作流串原子           │
└───────────────┬───────────────────────┬────────────────────┘
                │ 调用                    │ 调用
┌─ 领域逻辑原子 ▼──────────┐  ┌─ 领域逻辑原子 ▼─────────────┐
│ collect-plan 收集计划     │  │ kb-extract 抽取归类          │
│ collect-clean 清洗        │  │ kb-route 路由幂等+覆盖保护    │
│ collect-slot-fsm 状态机   │  │ kb-digest 周期周报           │
│ collect-nudge 群@催办+加急│  └─────────────────────────────┘
│ collect-wrapup 收工汇报   │
│ collect-store 任务/槽位库 │
└──────────────┬───────────┘
┌─ 飞书 I/O 能力原子（跨域复用）▼──────────────────────────────┐
│ feishu-message  feishu-contact  feishu-file   feishu-cell-write│
│ feishu-doc      feishu-minutes  feishu-chat   feishu-task      │
│ feishu-okr      feishu-comment                                 │
└──────────────────────────┬────────────────────────────────────┘
┌─ 共享底座 feishu-shared（lib，非用户可调）▼────────────────────┐
│ lark-exec(超时/重试/parse守护/exit-code) · idempotency(hash)   │
│ base-crud(分页 record-search / record_id 守护 upsert / get)    │
└───────────────────────────────────────────────────────────────┘
```

**调用方向单向向下**：编排器 → 领域原子 → I/O 原子 → 共享底座。原子之间不互相调用（除经由底座），保证可独立测试与复用。

## 四、编排层如何调用原子（数据流）

### 4.1 collector 编排（收集线）
```
@机器人+文件 → [编排器] on-message 归一
  → feishu-file.parse(链接/本地) ─┐
  → collect-plan(结构+指令) ───────┼→ 计划(槽位/责任人原文/场所/截止/催办)
  → feishu-contact.resolve(责任人) ┘   (none/ambiguous→消歧)
  → [确认闸1] 编排器发起确认卡片(LLM 复述)
  → collect-store.createTask/createSlots
  → feishu-message.send(首轮问) + collect-slot-fsm(→已问)
收集对象回复 → [编排器] 映射答案(LLM) → collect-clean → [确认闸2] → feishu-cell-write 写回 → collect-store.upsertValue(幂等) → fsm(→已填)
cron tick → collect-nudge(挑+群@发+加急) / collect-wrapup(过期→未交名单+汇报+置完成)
```

### 4.2 kb-maintainer 编排（知识库线）
```
会议事件 → [编排器] on-event 归一
  → feishu-minutes.fetchNotes → kb-extract.normalizeItems
  → kb-route.decide(查表/覆盖保护) → write? → feishu-doc.append(纪要) / feishu-task.create(行动项) / feishu-okr.createProgress(进展)
  → kb-route.record(回写指纹+落点)
群消息(累积窗) → kb-extract → kb-route → feishu-doc.update(群聊沉淀)
评论兜底 → feishu-comment.readDocBody/list → (LLM 理解) → feishu-comment.reply → kb-route.record(comment)
cron → kb-digest(遍历→汇编→feishu-doc 写周报)
```

每个原子 = 一个可 `node bin/<atom>.js ...` 或被 import 调用的单元，自带极简 SKILL.md（输入/输出契约）与单测。编排器的 SKILL.md 只描述"何时调哪个原子、怎么串"。

## 五、等价性保证策略（对外行为不变）

1. **纯搬移先行**：先把模块按上表搬进各原子（含已合并的 larkcli/hash/base-crud），保持函数签名与行为**逐字不变**，跑通现有 31 单测 + 各原子新单测，确认行为等价。
2. **修复随后单列**：等价搬移通过后，再在新原子上落 batch1+2 修复（每项一笔，单测覆盖）。修复是**有意行为变更**，逐条记录"原行为→新行为"，不混入搬移笔。
3. **对外契约不变**：两个编排器的对外触发方式（on-message/on-event/tick/digest 入口、环境变量、命令面）保持不变；Hermes/OpenClaw/CC 挂载方式不变。
4. **回归基线**：搬移阶段以"现有 34 单测全过 + 各 bin dry-run 输出不变"为等价判据。
