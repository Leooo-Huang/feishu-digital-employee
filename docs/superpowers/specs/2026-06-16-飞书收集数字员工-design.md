# 飞书「收集数字员工」· 设计

> 目标：给现有「公司知识库自动维护机器人」加一个**主动收集**型新技能——你上传一个文档/表格（里面有待收集的信息）、@机器人下达指令，它就在群里/私信主动对话收集，清洗、确认后整理写回对应文件，并自动催办、收工汇报。
> 与现有 6 项功能的关系：现有是**被动沉淀**（听会议/群消息→抽取→写文档）；本功能是**主动收集**（带目标驱动多轮对话→填表）。是"同一个数字员工的新技能"，复用现有身份/事件/Base 理念，不是第二套系统。
> 运行时定位：**可移植能力**——做成符合 agentskills.io 标准的 skill，Claude Code / OpenClaw / Hermes 都能用，不绑死任何一家。
> 编写日期：2026-06-16。基于 brainstorming 全程确认（见文末决策记录）。

## 0. 需求确认结果（来自与用户的逐项澄清）

| 维度 | 结论 |
|---|---|
| 收集形态 | **通用收集引擎**——表格按行收集 / 问题清单向特定人 / 开放式登记，三种都支持 |
| 收集场所 | **由任务决定**（发起时选）：群内公开 / 私信逐人 / 混合 |
| 问谁（责任人） | 四种来源都支持：文件责任人列 / 指令里 @点名 / 面向全群谁领谁填 / 按通讯录·群成员智能匹配（统一用 `lark-contact` 做人名解析器） |
| 文件形式 | 飞书在线表格·多维表格·文档 **与** 本地 Excel/Word **都支持**（本地文件内部先转飞书在线文档，再走同一套读写） |
| 数字员工行为 | 全要：自动追问/清洗不合格回答、带截止的定期催办、进度可随时查 + 收工自动汇报、写回前复述确认 |
| 运行时 | 不绑单一运行时；做成可移植 skill，三家通用（用户原话："openclaw hermes claude code 都可以用"） |

## 1. 四层可移植架构

核心思想：**"大脑/逻辑"做成可移植 skill，"手脚"用官方 lark-cli skills，"记忆"放飞书多维表格，只有"触发心跳"是各运行时一层薄适配。**

| 层 | 是什么 | 可移植性 |
|---|---|---|
| 🧠 **大脑（可移植 skill）** | `feishu-collector` skill（SKILL.md + 辅助脚本），编码整个收集工作流：解析文件→建收集计划→解析人名→对话收集→清洗/确认→写回→催办→汇报。**逻辑全在这里。** | ✅ Claude Code/OpenClaw/Hermes 原样通用（agentskills.io 标准） |
| ✋ **手脚（官方 lark-cli skills）** | `lark-drive`/`lark-markdown`(解析上传文件)、`lark-sheets`/`lark-base`/`lark-doc`(读结构+写回)、`lark-contact`(人名解析)、`lark-im`(群发/私信/加急)、`lark-event`(收消息) | ✅ 已装已授权，本身就是可移植 skill |
| 💾 **记忆（飞书多维表格 Base）** | 两张表：任务表 + 槽位表（见 §2）。复用现有设计"路由+幂等"理念。 | ✅ 状态存飞书云端，不绑任何运行时本地状态，**重启不丢、可在飞书里直接审计** |
| 💓 **心跳（各运行时薄适配）** | 同一 skill 的两个入口：① 来消息时推进对应任务 ② 定时 tick 时查截止/催办。靠 `lark-cli event consume` + 运行时调度器（OpenClaw cron / CC schedule / Hermes 外接 cron）。**只有这层各家不同，且很薄。** | ⚠️ 每家一份薄适配，上面三层完全一致 |

**让这套能移植 + 抗崩溃的关键原则**：skill 的每次调用都是**无状态的**——所有状态在 Base。每来一条消息或一次 tick = "从 Base 读任务状态 → 判断下一步 → 执行 → 写回 Base"。不依赖运行时内存/会话，所以既跨运行时又抗重启。

**手脚层选型**：主用 `lark-cli`（已装已授权、免常驻、最省事）；仅"实时事件订阅"一处，若某运行时需更强事件能力，可选接官方 `lark-openapi-mcp`——两者都是可移植标准，不冲突。

```
发起群/私信 ──消息──▶ [心跳:event consume] ──▶ feishu-collector skill ──▶ [手脚:lark-cli] ──▶ 飞书(消息/文件)
                                                      ▲   │
调度器(cron) ──tick──────────────────────────────────┘   ▼
                                                  [记忆: 飞书多维表格 Base]
```

## 2. 数据模型

核心抽象：**一个收集任务 = 一组"槽位（slot）"**。槽位 = `字段 × 关于谁/哪一项 × 写回到哪个具体位置 × 责任人(可选) × 值 × 状态`。三种形态都是这同一模型的特例。

### 2.1 任务表（一条 = 一次收集活动）

| 字段 | 含义 |
|---|---|
| `task_id` | 主键（自增/uuid） |
| `标题` | 如"Q3 团建报名信息收集" |
| `状态` | 草稿 / 待确认 / 收集中 / 暂停 / 已完成 / 已取消 |
| `场所` | 群 / 私信 / 混合 |
| `发起群chat_id` | 来源群 |
| `发起人` | open_id |
| `源文件` | token + 类型（sheet/base/doc/本地转换） |
| `目标文件` | 默认 = 源文件 |
| `截止时间` | deadline |
| `催办策略` | 如"每 24h，临近截止升级加急" |
| `原始指令` | @机器人 那句话原文（供 LLM 参考） |
| `收集计划摘要` | LLM 生成的人话版计划 |
| `created_at`/`updated_at`/`completed_at` | 时间戳 |

### 2.2 槽位表（一条 = 一个待收集信息点）

| 字段 | 含义 |
|---|---|
| `slot_id` | 主键 |
| `task_id` | 关联任务 |
| `字段名` | "身份证号" / "T恤尺码" / "预算" |
| `对象` | 这条关于谁/哪行（如"张三"）；开放式登记下为提交编号 |
| `责任人open_id` | 解析后的飞书用户（开放式可空） |
| `责任人原文` | 解析前文本（"张三"） |
| `落点` | 精确写回位置：sheet→单元格(如 C5) / base→record_id+字段 / doc→block 锚点 |
| `值` | 清洗后的值 |
| `状态` | 待问 / 已问 / 收到原始 / 清洗中 / 待确认 / 已填 / 跳过 / 不适用 / 待澄清 |
| `内容指纹` | content_hash，幂等用 |
| `追问次数` | attempts |
| `最近询问时间` | last_asked_at |
| `来源` | 消息 id / 链接，留痕 |

### 2.3 三种形态如何统一映射

| 形态 | 槽位何时建 | 责任人来自 | 写回方式 |
|---|---|---|---|
| 表格按行收集 | 解析时预建（行 × 可填列） | 行内人名 → `lark-contact` 解析 | 填回该单元格（原地 upsert） |
| 问题清单 / 向特定人 | 解析时预建（每问一槽） | 文件责任人列 / 指令 @点名 | 填回对应位置 |
| 开放式登记 | **提交时动态建**（每份上报生成一组槽） | 提交者本人（提交时捕获） | **追加新行** |

差异只有三处：槽位预建 vs 提交时建、责任人预绑 vs 提交时捕获、填格(原地) vs 追加(新行)。其余逻辑共用。

### 2.4 幂等与抗重启

写回 = 按"落点"做 **upsert**：写前比 `内容指纹`，没变 skip、变了才覆盖。每次 skill 调用从槽位表重新读状态，重启/重跑绝不重复写或填错格。

## 3. 对话流程与状态机

### 3.1 整体时序（以"私信逐人收集"为例）

**① 发起** — 你在群里 `@机器人 + 文件 + 指令`。心跳层捕获 → 调起 skill：
1. 解析文件结构（`lark-sheets`/`lark-base`/`lark-doc`）
2. LLM 生成**收集计划**：字段、责任人（`lark-contact` 解析人名）、场所、截止、催办策略
3. **发起确认**（写真人前第一道闸）：群里发卡片"我将私信收集 N 人的【字段…】，X 截止、到期催办，确认开始？"
4. 你确认 → 任务+槽位落 Base（状态=收集中）→ 群里公告

**② 收集** — 对每个有待填槽位的责任人按场所发问，槽位转 `已问`。对方回消息 → 心跳捕获 → skill 从 Base 载入此人相关槽位 → LLM：
- 把回答映射到槽位
- **清洗**：校验格式（身份证 18 位？尺码 ∈ {S,M,L,XL}？）→ 不合格/缺/歧义就追问，停在 `清洗中`
- 合格 → **复述确认**（"我记的是 张三 / 110… / L，对吗？"）→ OK → upsert 写回 + 记指纹 → `已填`
- 多人并行：每条消息独立推进各自槽位，靠 per-slot 状态 + 幂等防竞态

**③ 催办** — 调度器定时 tick → skill 载入任务 → 找出超过催办间隔、未 `已填` 的人 → 私信/@ 重问，临近截止升级**飞书加急**；`追问次数`+1。到期：最后一催 → 剩余按策略 `跳过` 并计入汇报。

**④ 进度 & 收工** — 你随时 `@机器人 进度？` → 回"已收齐 18/25，还差 张三、李四 的身份证号"。全部 `已填`（或到期）→ 群里发完成汇报 + 附填好的文件链接 → 任务 `完成`。

### 3.2 两级状态机

- **任务**：草稿 → 待确认 → 收集中 →（暂停）→ 完成 / 取消
- **槽位**：待问 → 已问 → 收到原始 → 清洗中 ⇄追问 → 待确认 ⇄改正 → 已填 ／ → 跳过·不适用·待澄清

### 3.3 消息归属判定

- 私信来的人若在某活跃任务有待填槽位 → 路由该任务；跨多任务 → LLM 反问消歧
- 群里 @机器人 的消息先由 LLM 分类意图：新指令 / 进度查询 / 开放式登记上报 / 暂停取消指令

## 4. 异常路径与边界

| # | 场景 | 处理 |
|---|---|---|
| 1 | 人名解析不到 / 重名 | `lark-contact` 模糊匹配返回 0 或多个 → 槽位标 `待澄清`；发起时把无法匹配的人名列给发起人确认，或群里 @所有人认领 |
| 2 | 一人对多任务 | DM 按"该人有待填槽位的活跃任务"路由；多个则 LLM 反问"你说的是【团建报名】还是【报销信息】？" |
| 3 | 超时 / 无人应答 | 到截止按策略：剩余槽位 `跳过`，汇报里列"未交名单"；**绝不编造填充** |
| 4 | 写回权限不足(403) | 写文档/表格需机器人是协作者+编辑权 → 捕获 403 → 通知发起人"请把我加为该文件协作者"；**不静默失败** |
| 5 | 群全量消息敏感权限 | 本功能**天然不需要** `im:message.group_msg`——只处理"@机器人的发起/进度消息"和"收集对象的定向回复(私信或@)"。**这是相对被动沉淀线的重要优势，绕开敏感审批。** |
| 6 | 文件中途被人改 | 写回前重读目标位置；若该格已被人工填且与将写入不一致 → 不覆盖，转 `待确认` 问发起人 |
| 7 | 加急合规 | 飞书加急(buzz)有频率/合规限制 → 默认先普通提醒、临近截止再升级加急，每人最多 N 次 |
| 8 | 重复提交 / 改答 | 开放式登记同一人重复提交 → 按任务级配置覆盖或新增；定向收集改答 → 覆盖原值 + 更新指纹 + 留痕 |
| 9 | 并发双击 / 快速重复 | 幂等 upsert + 槽位状态机天然防重 |
| 10 | LLM 误判 | "复述确认"这一步兜底；提供者说"不对" → 回 `清洗中` 重来 |
| 11 | 格式/多语言杂乱 | 清洗时归一化（尺码大小写、全角半角、中英文、日期格式） |
| 12 | 本地文件转换失败 | 转飞书在线文档失败 → 明确告知发起人失败原因，不静默退化（遵守全局"禁止静默 fallback"） |
| 13 | 截止后迟到回答 | 任务级策略：默认收下并补填 + 标注"迟交"；可配置为拒收 |

## 5. 测试策略

- **单元**：
  - 文件解析器（各种 sheet/base/doc/本地 xlsx 结构 → 槽位计划）
  - 人名解析器（重名 / 零匹配 / 别名 / @提及）
  - 清洗器（身份证 / 尺码 / 日期 / 邮箱 各种脏输入归一化）
  - 幂等 upsert（同值 skip、变值覆盖、并发）
- **状态机**：用沙箱任务跑全状态迁移（待问→…→已填、追问环、改正环、超时跳过）
- **集成（沙箱）**：测试群 + 测试多维表格 + 2-3 个测试账号（或用户+助手模拟），跑通"发起→确认→收集→清洗→确认→写回→催办→收工"完整闭环。**不在真实业务群、不 @ 真人。**
- **抗重启**：收集中途 kill 进程 → 重启 → 验证从 Base 恢复、不重复发问、不重复写
- **可移植验证**：同一 skill 分别在 Claude Code 和 OpenClaw 下各跑一遍沙箱闭环，确认行为一致
- **验收（契约式）**：每个收集形态一条端到端用例 + 截止/催办/确认/幂等各一条边界用例

## 6. 实施前需用户确认的事项（"等你醒来"清单）

2. **scope 确认**：需要 `base:app`(任务/槽位表)、`im:message`(收发/加急)、`im:chat`(群信息)、`contact:user`(人名解析)、`docx`/`sheets`(写回)、`drive`(本地文件转换上传)、`offline_access`。是否都已授权？缺的一次性补授（遵守 [[lark-cli-auth-batch-scopes]]）。**本功能不需要敏感的 `im:message.group_msg`。**

4. **运行时落地**：先在 Claude Code 下手动验证 skill 逻辑；要无人值守常驻时，再选 OpenClaw（已装、自带 cron）挂心跳层。

## 7. 决策记录（brainstorming 过程）

- 架构选型：用户最初选"基于 hermes agent"；助手实测发现 **Hermes 本地未安装、OpenClaw 已装在用且自带 cron**（更适合定时催办），并指出现有文档把 OpenClaw/Hermes 混为一谈。用户据此改判为"做成跨运行时可移植能力，三家都能用"。
- 据此确定：能力做成 agentskills.io 标准 skill（可移植正解，非将就），手脚用官方 lark-cli skills，记忆放飞书多维表格，心跳各运行时薄适配。
- 三种收集形态统一成"槽位"模型；写回用内容指纹幂等 upsert；写真人前发起确认 + 写回前复述确认双闸；催办用飞书加急且节制。
