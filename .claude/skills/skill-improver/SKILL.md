# Skill Improver — 技能自进化

## 概述

一个元技能（meta-skill）：输入是"发现的问题"，输出是"升级后的 skill 文件"。

当任何 skill 产出不符预期——遗漏了关键步骤、判断标准不准确、与其他 skill 衔接断裂——本技能负责：

1. **诊断**：精确定位是哪个 skill 的哪条规则/步骤出了问题
2. **修复**：修改该 skill 的 SKILL.md
3. **传播**：检查所有相关 skill，确保修复后不出现新矛盾
4. **留档**：记录每次迭代的原因和变更

核心原则：**skill 是活的，不是写完就不动的。每次犯错都是 skill 进化的机会。**

## 何时使用

- 执行某个 skill 后发现结果有漏洞（功能标了完成但实际不能用）
- 多个 skill 之间衔接断裂（A 的输出不满足 B 的输入要求）
- 用户指出 skill 流程缺少关键环节
- 用户说"improve skill"、"修复 skill"、"skill 有问题"、"技能迭代"

何时不使用：
- 问题出在代码实现，不是 skill 定义 → 直接修代码
- 需要创建全新的 skill → 用 `superpowers:writing-skills`
- 只是想了解某个 skill 怎么用 → 直接读 SKILL.md

## 流程

```
问题输入 → 证据收集 → 根因定位 → 影响分析 → 修复 skill → 一致性验证 → 记录迭代
```

## 详细步骤

### 步骤 1：证据收集

从 `$ARGUMENTS` 或对话上下文中提取：

| 信息 | 来源 | 示例 |
|------|------|------|
| **症状** | 用户描述或执行结果 | "hold-detect 标了 [~] 但实际 holds=[] 空数组" |
| **触发的 skill** | 对话上下文 | `autodev-sync` |
| **预期行为** | skill 的 SKILL.md 定义 | "标 [x] 意味着功能完整可用" |
| **实际行为** | 代码/运行结果 | "文件存在但核心逻辑是桩代码" |

**关键**：不要猜测，从实际证据出发。读相关代码、skill 文件、执行日志，确认问题确实存在。

### 步骤 2：根因定位

按以下层级逐级排查，找到**最深层**的原因：

```
表面症状
  └─ 哪个 skill 产出了错误结果？
       └─ 该 skill 的哪条规则/步骤导致了错误？
            └─ 这条规则为什么不够？是遗漏、模糊、还是错误？
                 └─ 这个遗漏是孤立的，还是反映了一类系统性缺陷？
```

**根因分类：**

| 类型 | 特征 | 示例 |
|------|------|------|
| **规则遗漏** | skill 没有覆盖某个场景 | sync 没有验证"代码文件存在 ≠ 功能可用" |
| **规则模糊** | 规则存在但判断标准不够精确 | "代码存在 + 测试存在 → [x]"，但没定义什么算"测试存在" |
| **规则错误** | 规则本身就是错的 | 标记标准允许把桩代码标为已完成 |
| **衔接断裂** | 两个 skill 之间的接口不匹配 | autodev 不调用 TDD，开发阶段没有验证 |
| **系统性缺陷** | 多个 skill 共享同一个错误假设 | 整个流水线假设"文件存在 = 功能完成" |

输出一段根因分析（3-5 句话），明确指出：
- 出问题的 skill 名称
- 出问题的具体章节/规则
- 缺陷类型
- 是否是系统性的（影响多个 skill）

### 步骤 3：影响分析

在修复之前，先评估波及范围：

1. **扫描所有 skill**：
   ```
   Glob: C:/Users/Leo23/.claude/skills/**/SKILL.md
   ```
   以及项目级 skill：
   ```
   Glob: .claude/skills/**/SKILL.md（如果存在）
   ```

2. **找出关联 skill**：哪些 skill 与出问题的 skill 有依赖关系？
   - **上游**：谁的输出是它的输入？
   - **下游**：它的输出是谁的输入？
   - **共享概念**：哪些 skill 使用了相同的判断标准或术语？

3. **评估修复的连锁影响**：
   - 修改 A 的输出格式 → B 的输入解析会不会坏？
   - 修改判断标准 → 其他使用相同标准的 skill 要不要一起改？
   - 新增步骤 → 会不会和已有步骤冲突？

输出一个影响列表：
```
直接修复: autodev-sync（修改步骤 2 判断标准）
连带更新: autodev（阶段 6 后增加验证环节）
无需更改: autodev-ideation, autodev-ui, autodev-api（不涉及验证逻辑）
```

### 步骤 4：设计修复方案

针对每个需要修改的 skill，设计具体的修改内容：

**修复原则：**

| 原则 | 说明 |
|------|------|
| **最小改动** | 只改必要的部分，不趁机"重构"整个 skill |
| **向后兼容** | 新规则不能让之前正确的行为变错 |
| **可验证** | 新增的规则必须有明确的判断标准，不能模糊 |
| **显式优于隐式** | 宁可多写一条明确的规则，也不要依赖"常识推理" |

**修复模板：**

```
## 修复 {skill 名称}

### 问题
{根因分析的结论}

### 修改内容
- 位置：{章节名 / 步骤编号}
- 原文：{引用原文}
- 改为：{新内容}
- 理由：{为什么这样改能解决问题}

### 新增内容（如需要）
- 位置：{插入在哪个章节之后}
- 内容：{新增的规则/步骤}
- 理由：{为什么需要新增}
```

将方案展示给用户确认后再执行修改。

### 步骤 5：执行修复

用 Edit 工具修改 SKILL.md 文件。

**修改顺序**：先修复核心 skill，再修复关联 skill。如果 A 依赖 B，先改 B。

**每次修改后**，重新读取修改后的文件，确认：
- 修改内容正确写入
- 没有破坏原有格式
- 新旧内容不矛盾

### 步骤 6：一致性验证

修复完所有文件后，做一次全局一致性检查：

1. **术语一致性**：同一个概念在不同 skill 中的名称和定义是否一致？
   - 例："已完成 [x]" 在所有 skill 中的判断标准是否统一？

2. **接口一致性**：上下游 skill 的输入输出格式是否匹配？
   - 例：A 输出的状态标记格式，B 能正确解析吗？

3. **逻辑一致性**：不同 skill 中的规则是否存在矛盾？
   - 例：A 说"文件存在即完成"，B 说"必须跑测试才算完成"

4. **反模式一致性**：修复引入的新行为有没有被某个 skill 的反模式禁止？

如果发现不一致，回到步骤 4 追加修复。

### 步骤 7：记录迭代

在 `C:/Users/Leo23/.claude/skills/skill-improver/changelog.md` 中追加记录：

```markdown
## {日期} — {简短标题}

**触发**: {什么问题触发了这次迭代}
**根因**: {根因分析结论}
**修复**:
- `{skill-name}/SKILL.md`: {修改摘要}
- `{skill-name}/SKILL.md`: {修改摘要}
**验证**: {一致性检查结果}
**教训**: {这次迭代揭示的一般性教训，供未来参考}
```

## 常见模式库

以下是已知的 skill 缺陷模式，遇到类似问题时可以快速匹配：

### 模式 1：存在性 ≠ 可用性

**症状**：文件/代码/端点存在，但功能实际不能用（桩代码、硬编码空值、未接通的管道）
**根因**：判断标准只检查"是否存在"，不检查"是否可用"
**修复方向**：在判断标准中增加功能性验证（运行测试、检查关键逻辑路径、验证数据流通）

### 模式 2：流水线断裂

**症状**：阶段 A 完成了，但阶段 B 拿不到 A 的输出，或者 A 和 B 之间缺少必要的中间步骤
**根因**：阶段间衔接只检查文件存在，不验证内容是否满足下游需求
**修复方向**：在阶段过渡处增加输入验证（内容检查而非仅存在性检查）

### 模式 3：标记乐观主义

**症状**：状态标记（[x] 已完成）比实际完成度更乐观
**根因**：标记标准过于宽松，缺少负面证据检查（桩代码检测、空数组检测、TODO 注释检测）
**修复方向**：增加"桩代码检测"规则——搜索 `= []`、`= null`、`// TODO`、`// stub`、mock 数据等模式，存在时降级为 [~]

### 模式 4：验证缺位

**症状**：流程中没有任何步骤验证产出是否真的可用
**根因**：流水线设计时假设每一步都会正确执行，没有考虑验证环节
**修复方向**：在关键节点插入验证步骤，调用现有的测试/验证 skill

### 模式 5：skill 孤岛

**症状**：相关的 skill 之间没有互相引用或协调
**根因**：每个 skill 独立编写，没有考虑与其他 skill 的关系
**修复方向**：在 skill 中明确声明依赖关系和接口约定

## 反模式

| 禁止 | 应该 |
|------|------|
| 不看证据就猜测根因 | 先读代码和 skill 文件，用事实说话 |
| 只修复触发问题的 skill，不检查关联 skill | 做完整的影响分析和一致性验证 |
| 大规模重写 skill | 最小改动原则，只改必要的部分 |
| 修复后不验证一致性 | 每次修复后做全局一致性检查 |
| 不记录迭代历史 | 每次修复都写 changelog |
| 修改 skill 的核心定位或职责 | 只修复规则/步骤级别的缺陷。如果需要改职责，那是创建新 skill |
| 用户没确认就直接改 SKILL.md | 先展示修复方案，用户确认后再执行 |
