# Skill Improver — 迭代记录

## 2026-04-14 — imagegen code-gen 排版适配

**触发**: code-gen 模板生成的图片像网页截图，内容挤在中间，不适合小红书竖屏阅读
**根因**: code-gen.md 的模板设计原则只有技术约束（CSS/字体/尺寸），完全缺失排版层面的规则。模板使用居中 flexbox 布局（桌面网页思维），没有竖屏均匀分布的概念。系统性缺陷——影响所有 5 个 HTML 模板。
**修复**:
- `imagegen/methods/code-gen.md`: 新增"排版适配检查"步骤（2.5），重写"模板设计原则"——均匀分布 + 信息密度上限 + 可读性标准 + 手写笔记风格
- `imagegen/templates/*.html`: 全部 5 个模板重写——`justify-content: space-evenly` 均匀分布 + Google Fonts 手写字体（ZCOOL KuaiLe + Ma Shan Zheng）+ 浅紫暖色渐变背景 + 大字号
**验证**: 5 个模板设计语言统一，code-gen.md 排版原则与所有模板一致
**教训**: 为特定平台生成的内容，排版适配规则必须和技术约束放在同等优先级。"能渲染出来"≠"适合目标平台"。布局方式必须是"均匀分布撑满画面"而非"居中"或"顶部堆叠"。

## 2026-03-10 — 优先级排序 ≠ 工作流顺序

**触发**: autodev 流水线生成的 UI 设计把"营销内容工厂"(W1) 排在"竞品调研"(W4) 前面。但实际业务流程中，必须先调研再做素材——调研产出是营销工厂的输入。此外营销工厂内部还内置了"竞品爆款采集"节点，与独立的调研场景功能重叠。

**根因**: 2 个 skill 缺少场景间依赖关系的处理：
- `autodev-ideation`：步骤 6 按综合分排 W 编号，但 W 编号被下游当作工作流顺序使用。综合分反映"值不值得做"，不反映"先做还是后做"——两个维度被混淆。
- `autodev-ui`：步骤 2（信息架构）直接使用 W 编号顺序构建导航，没有校验是否符合用户工作流的自然顺序。

**修复**:
- `autodev-ideation/SKILL.md`: 步骤 6 新增"场景依赖图"必须产出，明确场景间数据流关系；W 编号改为按依赖拓扑排序（工作流顺序），不是优先级排序
- `autodev-ui/SKILL.md`: 步骤 2 新增"场景排序规则"，要求读取 ideation 的场景依赖图，导航顺序按拓扑序排列；反模式表新增"用优先级排序当导航顺序"

**验证**: 术语一致（"场景依赖图"两个 skill 统一）、接口一致（ideation 输出依赖图 → ui 读取）、逻辑一致（W 编号含义统一为工作流顺序）。

**教训**: 编号系统的含义必须显式定义。W1 到底是"最重要的场景"还是"第一个要做的场景"？如果不说清楚，每个下游 skill 会按自己的理解去用。序号 = 顺序，分数 = 重要性，两者不能混用。

## 2026-03-05 — 验证缺位 + 标记乐观主义

**触发**: autodev-sync 把 hold-detect 标为 `[~]` 部分完成，但实际 `holds=[]` 硬编码空数组，功能完全不可用。autodev 流水线开发完直接交付，没有验证环节。

**根因**: 系统性缺陷，涉及 3 个 skill：
- `autodev-sync`：步骤 2 判断标准只检查文件/函数是否存在，不检测桩代码（模式 1：存在性 ≠ 可用性 + 模式 3：标记乐观主义）
- `autodev`：阶段 6→7 之间无验证环节（模式 4：验证缺位）
- `acceptance-testing`：功能完整但未被任何 skill 引用（模式 5：skill 孤岛）

**修复**:
- `autodev-sync/SKILL.md`: 步骤 2 新增"桩代码检测"必执行环节（5 种桩代码信号）；修正 `[~]` 定义，核心数据管道断裂应标 `[ ]`；反模式新增 2 条
- `autodev/SKILL.md`: 7 阶段 → 8 阶段，插入阶段 7"验证"（调用 `acceptance-testing`），含阻塞规则；更新流程图、state.yaml 模板、TodoWrite 条目、睡眠模式引用、反模式表格

**验证**: 一致性检查通过——术语、接口、逻辑、阶段编号均无矛盾。发现并修复了睡眠模式中残留的"阶段 7=交付"引用。

**教训**: 流水线设计时必须区分"存在"和"可用"。文件存在、代码存在、路由存在都不等于功能可用。每条验证链必须有至少一个"跑一遍"的环节，不能只靠静态检查。

## 2026-03-07 — autodev-iterate 文档同步从可选改为强制

**触发**: 用户指出 autodev-iterate 阶段 7 同步文档被跳过，文档没更新（YOLO 替代 HSV、新端点、颜色选择等改动都没同步到 docs/plans/）。用户明确要求"强制这一步"。

**根因**: `autodev-iterate` 阶段 7 标为"可选"（规则错误）。执行者跳过了同步，连"告知用户可以同步"都没做。代码改了但文档没更新，导致下次迭代阶段 1 读到过时信息。

**修复**:
- `autodev-iterate/SKILL.md`: 阶段 7 标题加"（强制）"，正文从"可选"改为"必须"+"不可跳过"；流程图 sync 节点加"（强制）"标签；反模式表新增"改完代码不同步文档"

**验证**: 一致性检查通过——流程图、标题、正文、反模式表四处均一致。其他 skill（autodev, autodev-sync 等）无需更改。

**教训**: 流水线中每个阶段的输出都是下一轮迭代某个阶段的输入。如果某个阶段"可选"，就等于允许输入链断裂。除非有明确的理由（如部署），否则流水线步骤应该默认强制。

## 2026-03-10 — MECE 分层原则缺失导致功能分类重叠

**触发**: 电商 AI agent ideation 输出中，"电商场景模板库"包含了"营销素材生成"，同时"营销内容工厂"又是独立 MVP 功能——分类违反 MECE 原则，平台能力和业务场景混在同一层级。

**根因**: `autodev-ideation` 的步骤 5（迭代检查）、步骤 6（功能排序）、步骤 7（文档模板）均没有分层指导。所有功能被当作扁平列表处理，没有区分"平台层共享能力"和"业务场景层端到端流程"。属于规则遗漏 + 系统性缺陷（4 个 skill 的衔接都基于扁平功能列表）。

**修复**:
- `autodev-ideation/SKILL.md`: 步骤 5 新增 MECE 分层检查（互斥性、穷尽性、无包含关系）+ 反模式示例；步骤 6 改为"先分层再排序"，引入 P/W 编号规则 + 依赖关系；步骤 7 文档模板改为"平台层 MVP + 业务场景层 MVP"双表格
- `autodev/SKILL.md`: 阶段 1→2 衔接更新，明确读取 P/W 编号和依赖关系，告知 brainstorming 按分层做架构设计
- `autodev-ui/SKILL.md`: 步骤 1 提取逻辑区分 P 编号（→ 共享 UI 组件）和 W 编号（→ 独立页面/工作流）+ 依赖关系
- `autodev-sync/SKILL.md`: 步骤 1 提取逻辑按层级分组追踪，P/W 前缀标签；新增依赖就绪检查规则

**验证**: 术语一致性（P/W 编号 4 个 skill 统一）、接口一致性（ideation 输出 → autodev/ui/sync 读取格式匹配）、逻辑一致性（sync 依赖降级规则与桩代码检测规则兼容）——全部通过。

**教训**: 功能清单不能是扁平列表。当产品有"平台 + 场景"的双层架构时，功能分类必须显式分层，否则必然出现重叠和包含关系。skill 应该内建分类框架指导，而不是依赖执行者的"常识"来做 MECE。

## 2026-03-13 — 需求展开缺失 + 功能迁移形态未评估

**触发**: 用户说"幽灵透明度移到设置，包括骨骼颜色等等"，结果只做了 ghostOpacity 一项；且直接内联到设置页，与现有 UI 风格不一致。

**根因**: `autodev-iterate` 3 处规则遗漏：
- 阶段 2c 代码分析员：无"需求展开"指导，不会在用户说"等等"时枚举所有同类项
- 阶段 3 综合分析：无"需求完整性 MECE 验证"，不检查是否覆盖了用户所有需求点
- 阶段 4 方案设计：无"功能迁移交互形态"决策指导，不评估内联/子页面/展开面板

**修复**:
- `autodev-iterate/SKILL.md`: 阶段 2c 新增第 5 项"需求展开"研究内容（含触发词和搜索指导）；阶段 3 新增第 0 步"需求完整性验证 MECE"（含停下确认规则）；阶段 4 方案必须包含新增第 4 项"功能迁移形态"（含 ≤3/4-8/>8 决策标准）；反模式表新增 2 条

**验证**: 三处修改逻辑连贯——2c 产出展开清单 → 3 验证清单完整性 → 4 基于完整清单评估交互形态。其他 skill 无需更改。

**教训**: 用户说"等等"不是修饰语，而是一个隐式需求——要求你去枚举所有相关项。AI 不能把开放式需求当作"只做提到的那一个"。同样，"移到某个页面"不只是"把代码复制过去"，还要评估目标页面的信息密度和交互一致性。

## 2026-03-10 — 前端代码质量断链：开发阶段不调用 frontend-design

**触发**: 用户反馈 autodev 生成的前端界面"经常很丑"。frontend-design 插件有完整的美学指导，但流水线中从未引用它。

**根因**: 4 个 gap 导致设计→代码质量断链：
1. `autodev` 阶段 6 不知道要用 `frontend-design` skill（skill 孤岛）
2. `autodev` 阶段 6 subagent 可能不读 `*-ui.md`（上下文丢失）
3. 无处声明前端技术栈偏好（Next.js/shadcn 等）
4. `autodev-iterate` 迭代前端时同样不调 `frontend-design`

**关键决策**：用户最初要求"规范成 Next.js"写入 skill，经分析后否决——技术栈是项目偏好，不是 skill 规则。正确位置是 config.yaml 的可选配置。

**修复**:
- `autodev/SKILL.md`: 阶段 6 新增"前端代码质量规则"——前端任务必须传入 ui.md + 通过 frontend-design skill 生成代码 + 读取 config 前端偏好；config.yaml 模板新增 `phases.frontend` 可选配置节
- `autodev-iterate/SKILL.md`: 阶段 5 实现规则新增"涉及前端 UI 改动时必须通过 frontend-design skill"

**验证**: autodev-ui 和 frontend-design 职责无重叠（结构设计 vs 美学代码）；config.yaml 前端偏好是可选的，不影响无前端项目。

**教训**: 有好的 skill 但不在流水线中引用 = 等于没有。skill 孤岛问题不只是"没人调用"，更深层是编排器（autodev）没有为每个阶段声明"可用的专业 skill"。

## 2026-03-18 — autodev-iterate 频繁跳过阶段 2-3（研究团队+综合分析）

**触发**: 用户多次反馈 autodev-iterate 跳过流程，直接从读代码跳到实现。用户原话："iterate不可以略过流程"。

**根因**: 3 处规则缺陷导致模型自行判断"可以跳过"：
1. 阶段 2 写了"至少 2 个"但没写"不可跳过"，模型判断简单改动不需要研究团队
2. 阶段 4 有"简单修复→直接实现"逃生口，模型把大部分修改判断为"简单"
3. 阶段间没有强制门控输出标记，模型可以随意跳阶段

**修复**:
- `autodev-iterate/SKILL.md`:
  - 单条流程前新增 `<IMPORTANT>` 强制声明：绝对不可跳过任何阶段 + 违反检测规则（未调 Agent 就 Edit = 违规）
  - 阶段 4 去掉"直接实现"逃生口，改为"简要方案展示给用户后实现"
  - 所有 7 个阶段末尾增加 `--- 阶段 N 完成 ---` 门控输出标记

**验证**: 一致性检查通过。其他 skill 无需更改。

**教训**: "必须"和"应该"在 skill 中效果完全不同。"必须用 Agent 创建研究团队"会被理解为"推荐"；只有 `<IMPORTANT>` + 明确的违反检测规则（"如果你没调 Agent 就开始 Edit = 违规"）才能真正阻止跳过。规则的强制性需要通过"如何检测违反"来体现，而不只是声明"不可跳过"。

## 2026-03-30 — 文档约束体系升级：索引层+规则层+漂移检测+设计修正+Hooks推荐

**触发**: last30days 研究发现社区在 spec-driven development 方面形成共识（GitHub spec-kit 72.7k stars, @akshay_pachaar 972 likes 等）：大模型上下文越长越容易丢失约束，需要索引文档层、规则文档层、hooks 确定性约束、文档漂移检测。当前 autodev 系列 skill 全部缺少这些机制。

**根因**: 5 个系统性缺陷：
1. 缺少索引文档层（Index Layer）：5 个设计文档无总入口，开发 subagent 被迫加载全部文档导致约束稀释
2. 缺少独立规则文档层（Rules Layer）：编码规则散落在 CLAUDE.md、SKILL.md、各 skill 反模式中，subagent 无精炼的 do's/don'ts
3. 文档漂移检测缺失：autodev-sync 只检查完成状态，不检查设计决策是否仍有效
4. Hooks 集成缺位：质量红线全部是指令性软约束，无确定性硬约束
5. 文档不能自演化：实现偏离设计文档时无机制回头修正

**修复**:
- `autodev/SKILL.md`:
  - 新增 Phase 5.5：生成 `*-index.md`（< 100 行项目全景索引）和 `*-rules.md`（< 80 行编码规则手册）
  - Phase 6 改为 Token 效率优先的上下文加载策略：subagent 先读 index+rules，按需读详细文档
  - Phase 8 新增 hooks 推荐步骤：从 rules.md 生成 Claude Code hooks 配置建议
  - 更新阶段表、文档链图、红线执行时机表
- `autodev-sync/SKILL.md`:
  - 新增步骤 2.5：设计漂移检测（技术选型/API端点/数据模型/页面路由 5 维度，INFO/WARN/ERROR 3 级）
  - 同步报告模板新增漂移章节和文档摘要一致性检测
  - 反模式表新增"只检查完成状态不检查设计漂移"
- `autodev-iterate/SKILL.md`:
  - 阶段 7 升级为"同步文档 + 设计修正"：7a 设计漂移修正（含检查清单+修正格式+连带更新 index/rules）→ 7b 状态同步
  - 阶段 1 新增读取 index.md 和 rules.md
  - 反模式表新增 2 条（不修正偏离文档、忘更新摘要文档）

**验证**: 跨文件一致性检查发现 5 个问题并全部修复：
1. 术语统一："设计修正" → "设计漂移修正"
2. autodev-sync 中 index/rules 文件加中文描述名
3. 同步报告模板补充 index/rules 一致性输出
4. autodev-iterate 阶段 1 补充 index/rules 文件说明
5. autodev-sync 步骤 2.5 补充执行时机说明

**教训**: 文档体系需要像代码一样有"索引+模块"的分层结构。把所有设计文档塞进 subagent 的 prompt = 把整个代码库塞进一个函数。索引文档是"头文件"，规则文档是"lint 配置"，hooks 是"CI 管道"——三者缺一不可，分别提供软约束、精炼约束、硬约束三层保障。

## 2026-04-12 — 文档同步前置化（阶段 4.5 预更新）

**触发**: 用户多次指出 autodev-iterate 完成代码后忘记同步文档（阶段 7 经常被跳过）。本次对话中 W6i-W6l 实现后、快捷回复功能改造后、认证系统改造后都忘了同步。

**修复**:

- `autodev-iterate/steps/04-design-solution.md`: 末尾新增"阶段 4.5：设计文档预更新"章节，含检查表（什么条件→更新什么文档）
- `autodev-iterate/steps/07-sync-and-fix-docs.md`: 从"完整文档同步+设计修正"简化为"状态确认"（内容已在 4.5 预更新，7 只做状态标记+微调检查）
