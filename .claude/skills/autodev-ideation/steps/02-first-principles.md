# 步骤 1.5：第一性原理拆解 — 定义本质任务

<IMPORTANT>
这一步在调研之前执行。目的是防止"直接跳到找功能"而遗漏不明显但关键的系统环节。
不做这一步，后续功能清单容易只覆盖"显而易见的需求"，而漏掉价值判断、风险过滤、反馈闭环等隐性环节。
</IMPORTANT>

**从用户想法出发，回答以下 5 个问题（每个问题 1-3 句话）：**

## Q1：本质任务
> 这个系统本质上在做什么变换？（不是功能描述，是一句话的本质）

写成：`输入 → [核心变换] → 输出` 的形式。

**例：**
- "GitHub 推送自动发社交媒体" → `代码变化 → [转译成适合传播的内容] → 多平台发布`
- "攀岩训练 app" → `用户的攀爬行为 → [量化分析 + 科学训练] → 可衡量的能力提升`
- "AI 工具选择器" → `需求清单 + 候选工具 → [约束优化] → 最优工具组合`

## Q2：核心价值链
> 从输入到输出，中间必须经过哪些**不可跳过**的环节？

列出 4-7 个环节，每个环节用一个动词短语描述。这些是系统的"骨架"——缺任何一个，系统就不完整。

**例（GitHub → 社交媒体）：**
1. 变化捕获（发生了什么更新）
2. 价值判断（这次更新值不值得发）
3. 内容转译（技术变化 → 人能理解的内容）
4. 平台适配（同一内容 → 不同平台的格式/风格）
5. 发布执行（调 API 发出去）
6. 质量控制（不发错、不泄密、不重复）

## Q2.5：用户旅程（User Journey）
> 用户从第一次接触到持续使用，完整经历了哪些步骤？

Q2 是系统内部视角（系统必须做什么），这一问是用户外部视角（用户经历了什么）。两个视角交叉对照，能发现单独一个视角看不到的遗漏。

列出用户从头到尾的完整步骤链，包括：
- **首次使用**的配置/连接步骤（往往被忽略）
- **核心操作**的每一步
- **操作后**的反馈/确认/查看结果步骤（也容易被忽略）

**例（GitHub → 社交媒体）：**
1. 连接 GitHub 账号（授权 OAuth）
2. 选择要监控的仓库
3. 配置发布规则（什么事件触发、发到哪些平台、审核模式）
4. 收到候选内容通知
5. 审核/编辑内容（或自动发布）
6. 内容发布到各平台
7. 查看发布效果（engagement 数据）
8. 调整规则/优化内容策略

**交叉对照：**

把用户旅程的每一步和 Q2 核心价值链对照：
- 用户旅程中有，但 Q2 没提到的步骤 → 可能是被忽略的系统能力（如"配置规则"、"查看效果"）
- Q2 中有，但用户旅程中不可见的环节 → 是后台能力，用户不直接接触但系统必须做（如"风险过滤"）

两种遗漏都要补到功能清单里。

## Q3：失败模式
> 这个系统最可能在哪里失败？列出 3-5 个"如果 X 没做好，整个系统就废了"的场景。

这些失败模式会直接转化为后续的必需功能。

**例：**
- 每次 push 都发 → 变 spam，用户取关
- 发布内容包含内部信息 → 安全事故
- 所有平台发同样的内容 → engagement 低

## Q4：成功标准
> 用户用了这个系统后，怎么判断它"好用"？列出 2-3 个可观测的成功指标。

不需要精确数字，要的是方向。

**例：**
- "发布的内容 engagement 比手动写的不差"
- "配置一次后，不需要每天人工干预"

## Q5：与"直觉方案"的差距
> 最容易想到的实现方案是什么？这个直觉方案会漏掉 Q2 中的哪些环节？

这一问专门用来暴露盲点。

**例：**
- 直觉方案："GitHub webhook + LLM 总结 + 发帖 API"
- 漏掉了：价值判断（不是每次 push 都值得发）、平台适配（不同平台要不同风格）、质量控制（防泄密）

---

**产出格式：**

在后续的 ideation 文档中，新增一个 `## 本质任务分析` 章节（位于"产品定位"之前），包含以上 5 个问题的回答。

**这 6 个回答如何影响后续步骤：**
- Q2（核心价值链）→ 直接成为步骤 3 功能发现的"必须覆盖"检查清单
- Q2.5（用户旅程）→ 与 Q2 交叉对照，发现被系统视角遗漏的功能（如配置、反馈）；同时传递给阶段 3（UI/UX）作为页面流设计的输入
- Q3（失败模式）→ 直接转化为功能需求（每个失败模式对应至少一个防护功能）
- Q5（直觉方案差距）→ 在步骤 5 迭代检查时，验证功能清单是否补上了这些盲点
