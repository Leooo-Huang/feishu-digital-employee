# 步骤 1：读取现有设计文档 + 代码资源扫描

**目标**：理解当前项目的完整架构和功能版图，并发现可复用的代码资源，避免重复开发。

## 执行

### 1a. 读取设计文档

1. **读取精炼摘要**（优先）：
   - `*-index.md` — 项目全景、关键决策、能力-组件映射
   - `*-rules.md` — 编码约束、API 约定

2. **读取功能清单**：
   - `*-ideation.md` — 现有 MVP 功能列表（P/W 编号）、场景依赖图
   - 记录：当前有哪些功能、哪些已完成、哪些在进行中

3. **读取架构**：
   - `*-design.md` — 技术栈、架构模式、组件划分、能力-组件映射表
   - 记录：新功能应该放在哪个组件/模块下，还是需要新建组件

### 1b. 代码资源扫描（关键——避免重复造轮子）

4. **扫描数据模型**：
   - 读取 Prisma schema / TypeORM entities / 数据库 migration 文件
   - 列出所有现有表/模型及其字段
   - 标注：哪些表可能与新功能相关？（如新功能涉及"报告"，现有是否已有 `reports` 表？）

5. **扫描 API 端点**：
   - Grep 路由注册文件（`app/api/`、`routes/` 等）
   - 列出所有现有端点（方法 + 路径）
   - 标注：哪些端点可以直接复用或扩展？

6. **扫描 UI 组件**：
   - Glob `components/` 目录，列出现有组件
   - 标注：哪些组件可以在新功能中复用？（如已有 `DataTable`、`Modal`、`FileUpload`）

7. **扫描工具函数和服务**：
   - 检查 `lib/`、`utils/`、`services/` 目录
   - 标注：哪些服务可以复用？（如已有 `StorageService`、`ExportService`、`NotificationService`）

8. **快速扫描代码现状**：
   - `git log --oneline -20` — 最近的开发方向
   - 目录结构 — 现有的文件组织方式

## 必须输出

向用户展示：

**项目概况**：
- "当前项目有 N 个功能：{列表}"
- "架构是 {技术栈}，{架构模式}"
- "你要添加的新功能是：{用户输入}"

**可复用资源清单**：
- "数据模型：{可复用的表/模型列表}"
- "API 端点：{可扩展的端点列表}"
- "UI 组件：{可复用的组件列表}"
- "工具/服务：{可复用的函数/服务列表}"

**初步判断**：
- "新功能应该 {融入现有模块 X / 新建独立模块}"
- "预计可复用 {N} 个现有资源，需要新建 {M} 个"
