# 步骤 2：完整深度需求分析

**目标**：对新功能做完整深度的需求分析——不是"轻量"的，而是和 `/autodev-ideation` 同等深度。一个用户需求往往由多个子功能组成，必须在设计前全部发现。

## 执行

### 2a. 第一性原理分析

对新功能执行 Q1-Q5 分析（参考 `autodev-ideation/steps/02-first-principles.md`）：

- **Q1**：用户的**根本目标**是什么？（不是"生成报告"，而是"基于数据做出更好的决策"）
- **Q2**：完成这个目标的**完整价值链**是什么？（数据采集→分析→报告生成→报告查看→数据溯源→报告管理→报告分享）
- **Q3**：每个环节如果**出错/缺失**会怎样？（生成了但找不到历史→丢失价值；数据不可追溯→用户不信任）
- **Q4**：有哪些**隐含假设**？（用户只生成一次？还是会反复生成？）
- **Q5**：**最小可用**是什么？（哪些子功能少一个就不能用？）

### 2b. 用户旅程展开

从用户视角走完整个旅程——**使用前、使用中、使用后**：

| 阶段 | 用户在做什么 | 需要什么功能 |
|------|------------|------------|
| **使用前** | 找到入口、了解功能 | 导航入口、功能说明 |
| **使用中（核心）** | 执行核心操作 | 主要功能（用户明确说的那个） |
| **使用中（辅助）** | 查看细节、验证数据 | 数据溯源、详情展开、筛选排序 |
| **使用后（管理）** | 回看历史、整理 | 历史列表、搜索、删除、归档 |
| **使用后（延伸）** | 分享、导出、协作 | 导出 PDF、分享链接、协作批注 |

对每个阶段问："如果这个阶段没有对应功能，用户会怎样？"——如果答案是"体验严重受损"，则该功能是必需的。

### 2c. 功能组发现

从 2a 和 2b 中提取完整的功能组：

```
核心功能：用户明确要求的功能
配套功能：不说不知道，但少了就不好用的功能（如历史管理、数据溯源）
防御功能：防止出错的功能（如确认弹窗、撤销操作、数据备份）
增强功能：有了更好但 MVP 可以不做的功能（如分享、协作）
```

### 2d. 与可复用资源交叉比对

把功能组中的每个子功能与步骤 1 的"可复用资源清单"对照：

| 子功能 | 有现成实现吗？ | 处理方式 |
|--------|-------------|---------|
| {子功能 A} | 有——`ExistingService.method()` | 直接复用 |
| {子功能 B} | 部分有——`ExistingTable` 有相关字段 | 扩展现有 |
| {子功能 C} | 没有 | 新建 |

### 2e. 确定范围和编号

1. **MVP 范围**：核心功能 + 必需的配套功能（少了体验严重受损的）
2. **V2 范围**：增强功能
3. **P/W 编号**：读取现有 ideation.md 的最大编号，分配新编号
4. **MECE 检查**：参考 `autodev-ideation/checklists/mece-check.md`

### 2f. 调研（按需）

- 竞品有类似功能吗？怎么做的？
- 涉及新技术时 WebSearch 验证可行性
- 不需要像 autodev-ideation 那样做完整的市场/前沿调研（因为产品方向已定）

## 必须输出

向用户展示：

**功能组分解**：
```
新功能：{用户原始需求}
  ├── 核心：{子功能列表}
  ├── 配套：{子功能列表}（少了会导致 {问题}）
  ├── 防御：{子功能列表}
  └── 增强（V2）：{子功能列表}
```

**MVP 范围确认**：
- "本次实现 {N} 个子功能（核心 + 配套）"
- "延后 {M} 个增强功能到 V2"
- "可复用 {K} 个现有资源"

**等待用户确认范围后再继续。**
