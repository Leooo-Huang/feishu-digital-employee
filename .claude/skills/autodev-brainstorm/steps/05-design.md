# 步骤 5：DESIGN — 详细设计

对选定方案进行深化设计，覆盖以下内容：

1. **架构概览**（含 ASCII 图，如有帮助）
2. **组件分解**（每个组件的职责、接口、依赖）
3. **数据流**（数据怎么流转，状态存在哪里）
4. **错误处理策略**（异常路径、降级方案）
5. **测试策略**（测试什么、怎么测、覆盖范围）
6. **风险与缓解**（基于调研中的"常见陷阱"章节）
7. **技术选型表**（核心依赖版本验证）

## 技术鲜度验证（必须在写入设计文档前完成）

<IMPORTANT>
对设计方案中涉及的每个核心依赖（框架、UI 库、ORM、状态管理、构建工具等），必须用 WebSearch 验证：
1. 当前最新稳定版本号
2. 是否有已废弃的 API 被方案引用
3. 近 6 个月内是否出现了更好的主流替代品

**不允许仅凭记忆中的知识做技术选型。** 训练数据有截止日期，库的版本和最佳实践在持续更新。

验证结果写入设计文档的「技术选型」章节：
```
| 依赖 | 选用版本 | 最新稳定版 | 查证日期 | 选型理由 |
```

如果 WebSearch 发现记忆中的版本已过时，以搜索结果为准。
如果发现更好的替代品且调研匹配度更高，应重新评估方案选择。
</IMPORTANT>

**复杂度匹配原则：** 简单功能写 1 页，复杂系统写 3-5 页。不要过度设计简单功能，也不要草草对付复杂系统。
