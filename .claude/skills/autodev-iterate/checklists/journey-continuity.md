# 用户旅程连续性检查

回到阶段 1 定位的用户旅程上下文，逐项检查：

## 上游衔接
- [ ] 从上一步过来的数据/状态，改动后还能正确接收吗？
- [ ] 上游页面的跳转链接/按钮是否仍然指向正确的位置？

## 下游传递
- [ ] 这一步的输出，下一步还能正常消费吗？
- [ ] 数据格式/字段名是否与下游期望一致？

## 旁路影响
- [ ] 如果旅程中有分支路径（如"审核通过"vs"审核拒绝"），两条路径都还正常吗？
- [ ] 错误/异常路径的处理是否仍然正确？

## 常见断链模式

| 改了什么 | 容易断在哪 |
|---------|-----------|
| 修改了数据结构/字段 | 下游页面读不到改后的字段名 |
| 修改了状态流转逻辑 | 下游依赖的状态值变了 |
| 修改了页面路由/入口 | 上游的跳转链接失效 |
| 修改了 API 返回格式 | 所有消费这个 API 的页面 |

如果改动是纯后台/纯算法，不影响用户可见行为，可以跳过此检查并说明原因。
