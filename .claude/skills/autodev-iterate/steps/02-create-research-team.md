# 阶段 2：创建研究团队（三路并行）

**必须用 Agent 工具创建并行研究代理。** 不要自己做所有分析——创建团队分头研究。

## 分派规则

根据阶段 1 的分类，选择创建哪些研究员：

| 分类 | 创建哪些 agent | 读取哪些文档 | 跳过什么 |
|------|---------------|-------------|---------|
| **UI** | UX 研究员 + 代码分析员 | ui.md + index.md | 技术研究员 |
| **样式** | 跳过全部研究 → 直接进阶段 4 | ui.md 视觉/动效章节 | 全部研究员 |
| **数据** | 技术研究员 + 代码分析员 | api.md + design.md | UX 研究员 |
| **API** | 技术研究员 + 代码分析员 | api.md | UX 研究员 |
| **逻辑** | 技术研究员 + 代码分析员 | design.md | UX 研究员 |
| **全栈** | 全部 3 个 | index.md + 按需 | 无 |

**样式类特殊处理**：
样式调整不需要研究团队。直接跳到阶段 4（设计方案），方案就是"改什么 CSS/token"。然后阶段 5 实现，阶段 6 用视觉验证循环。

**跳过研究员时的说明**：
输出："改动类型为 {类型}，跳过 {agent 名} 研究员（该维度与本次改动无关）。"

根据问题涉及的维度，创建以下研究员（按分派规则选择性激活）：

## 2a. UX 研究员（分类为 UI 或全栈时激活）

**职责**：从用户体验角度分析问题。

研究内容：
1. **竞品交互**：同类 App（如 Stokt、Kilter Board、MoonBoard）怎么做这个交互？
2. **平台惯例**：iOS/Android 原生手势惯例是什么？（拖拽移动、捏合缩放等）
3. **现有交互审查**：当前实现的交互流程是否符合用户直觉？
4. **设计原则检查**：是否违反了 UX-DESIGN.md 中的核心原则？

**输出格式**：
```
当前交互: {描述现有交互流程}
问题: {为什么不好用，从用户视角}
竞品做法: {至少 2 个竞品的做法}
平台惯例: {iOS/Android 对这类操作的标准手势}
建议交互: {具体的交互方案，包括手势、反馈、状态变化}
```

**UX 审查清单**（研究员必须逐项回答，详见 `checklists/ux-audit.md`）。

## 2b. 技术研究员（分类为数据、API、逻辑或全栈时激活）

**职责**：评估技术方案的可行性和最优解。

研究内容：
1. **当前技术方案评估**：现在用的算法/库/方案的局限性是什么？
2. **替代方案调研**：有没有更好的技术方案？（WebSearch 搜索最新实践）
3. **平台限制**：目标平台（如 React Native、Expo）对某些方案有已知限制吗？
4. **性能考量**：方案在目标设备上的性能表现如何？
5. **版本与时效性验证**：当前代码使用的库/API 是否仍是最新？有没有 deprecated？

<IMPORTANT>
**技术鲜度验证（必须执行）**：
- 对涉及的每个核心库/框架，WebSearch 验证当前最新稳定版本
- 对涉及的每个 API 调用，WebSearch 验证是否已被废弃或有更新的替代
- **不允许仅凭记忆推荐方案**——训练数据有截止日期，必须搜索确认
- 如果发现当前代码使用的库版本已过时且新版有更好方案，在输出中标注

搜索示例：
- `"{库名} latest stable version 2026"`
- `"{API名} deprecated replacement"`
- `"{技术方案} vs alternatives {当前年份}"`
</IMPORTANT>

**输出格式**：
```
当前方案: {描述现有技术实现}
局限性: {为什么效果不好}
版本状态: {当前使用版本} → {最新稳定版}（是否需要升级）
替代方案:
  方案A: {描述} — 优: {优点} / 劣: {缺点}
  方案B: {描述} — 优: {优点} / 劣: {缺点}
平台限制: {已知的兼容性/限制问题}
推荐方案: {选择哪个，为什么}（方案来源: WebSearch 查证日期 YYYY-MM-DD）
```

## 2c. 代码分析员（除样式类外，所有分类均激活）

**职责**：追踪代码数据流，定位问题根因。

研究内容：
1. **数据流追踪**：从用户入口 → 数据源 → 渲染，完整追踪
2. **对比文档和代码**：找到文档描述和实际实现的差距
3. **根因定位**：不要停在表面症状，往下挖一层
4. **关联影响**：修复这个问题会影响哪些其他功能？

**常见根因模式**：
| 模式 | 示例 |
|------|------|
| 桩代码未替换 | `holds = []` 硬编码空数组 |
| API 未对接 | 函数存在但未调用后端 |
| 平台兼容性 | SVG onPress 在 RN 不工作 |
| 数据流断裂 | Store 有数据但页面没读取 |
| 参数不适配 | 算法参数对当前场景不适用 |
| 错误的技术选型 | 用 HSV 分割做需要 ML 的工作 |

**5. 需求展开**（当用户需求含开放式列举时）：
   - 识别用户需求中的开放式词汇：「等等」「包括X」「类似的」「之类的」「etc.」
   - 搜索代码中**所有**与该类别相关的配置项/参数/状态，列出完整清单
   - 不能只处理用户显式提到的那一个，必须枚举同类项让用户确认范围
   - 示例：用户说"透明度移到设置，包括骨骼颜色等等" → 搜索所有可视化相关的可配置项（透明度、颜色、线宽、显示模式等），列出完整清单

**必须输出**：明确的根因列表（不是症状列表），涉及的文件清单，以及展开后的完整需求项清单（如有开放式需求）。
