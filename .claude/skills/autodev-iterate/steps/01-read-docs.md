# 阶段 1：读文档（理解设计意图 + 定位用户旅程）

在碰代码之前，先读相关设计文档：

1. 读 `docs/plans/` 下的设计文档（ideation、design、ui、api、plan）
2. 读 `CLAUDE.md`（项目级规范）
3. 读相关的 UX 文档（如 `mobile/docs/UX-DESIGN.md`）
4. 读 `*-index.md`（如存在，项目全景索引）
5. 读 `*-rules.md`（如存在，编码规则手册）
   （这两个文件由 autodev Phase 5.5 生成：index.md 包含项目全景概览和关键决策，rules.md 包含编码约定和质量红线。如果项目已执行完整 autodev 流水线，这两个文件会存在）

**目标**：理解这个功能**应该**怎么工作，而不是**现在**怎么工作。

## 自动分类

分析用户描述的改动，判断属于哪个类型：

| 类型 | 信号词/特征 | 例子 |
|------|-----------|------|
| **UI** | 页面、布局、组件、交互、流程、导航 | "工单列表加个筛选器"、"改一下详情页的布局" |
| **样式** | 颜色、字体、间距、动画、CSS、主题、配色 | "主色改成蓝色"、"间距太大了"、"加个 hover 效果" |
| **数据** | 数据库、schema、migration、字段、表、模型 | "用户表加个 phone 字段"、"改成软删除" |
| **API** | 端点、接口、请求、响应、认证、权限 | "加一个批量删除接口"、"返回值加个 total 字段" |
| **逻辑** | 算法、性能、计算、规则、后端处理 | "搜索太慢了"、"价格计算有 bug" |
| **全栈** | 涉及多个类别，或无法归入单一类别 | "加一个导出功能"（需要 API + UI + 逻辑） |

**分类规则**：
- 如果改动明确只涉及一个类别 → 归为该类别
- 如果涉及 2 个类别 → 取主要类别（用户最关心的那个）
- 如果涉及 3 个及以上 → 归为全栈
- 不确定时 → 归为全栈（宁可多做不漏做）

**分类决定了后续步骤的行为**（见步骤 2 的分派规则）。

**用户旅程定位（关键步骤）：**

从 ideation 文档的「本质任务分析 → 用户旅程」章节（如有），或 ui 文档的「用户流程」章节，找到**要修改的功能在用户旅程中的位置**：

- 这个功能在旅程的**第几步**？
- 它的**上游**是什么？（用户从哪来？带着什么上下文/数据？）
- 它的**下游**是什么？（用户做完这步之后去哪？传递什么数据/状态？）

这三个信息决定了：改动不能只看这个功能本身，还要验证上下游不断链。

**必须输出**：
- 用 1-3 句话总结"文档说应该是 X，用户反馈实际是 Y"
- 用户旅程上下文："这个功能是旅程中的第 N 步，上游是 {A}，下游是 {B}"
- 改动分类：{UI / 样式 / 数据 / API / 逻辑 / 全栈}
- 将读取的设计文档：{列出}
- 将创建的研究 agent：{列出}
