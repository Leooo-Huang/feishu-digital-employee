# 阶段 5：实现

按照方案改代码。可根据任务规模选择执行方式：

- **单人执行**：简单修复，直接改
- **团队执行**：多文件/多模块改动时，创建实现团队并行开发

**规则**：
- 只改必要的文件，不顺手重构
- 遵循项目的代码规范（如 `makeStyles(c: Theme)` 模式）
- 每完成一个改动点，更新 TodoWrite 进度
- **涉及前端 UI 改动时**：必须通过 `frontend-design` skill 生成/修改用户可见的页面和组件代码，确保视觉质量。同时读取 `*-ui.md` 中该页面的设计作为结构约束

## 样式类视觉迭代循环

当分类为**样式**时，实现阶段使用视觉循环而非常规实现：

```
循环（最多 3 轮）：
  1. 实现调整（修改 CSS/token/组件样式）
  2. 视觉验证：
     - 如果有 MCP 截图工具 → 截图并评估
     - 如果没有 → 描述改动效果，请用户在浏览器中确认
  3. 评估：
     - 符合视觉规范（ui.md 的视觉规范章节）？
     - 与现有页面风格一致？
     - 用户满意？
     ├─ 满意 → 结束循环
     └─ 不满意 → 记录问题，回到 1
```

**非样式类**改动走常规实现规则。

## GAN 式对抗审查

实现完成后，调用 `autodev-review` skill。

传入参数：
- 维度 3 = "最小改动"（autodev-iterate 特有，检查是否只改了必要的部分）
- acceptance_criteria 从 bug 描述/改进需求推导

`autodev-review` 会自动执行 4 维度打分 → 迭代循环 → pivot → 最多 5 轮。
