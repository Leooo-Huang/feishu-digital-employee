# 阶段 6：验证

**目标**：验证改动解决了问题且不引入新问题。

## 执行

调用 `autodev-verify` skill。

根据阶段 1 的分类，`autodev-verify` 会自适应选择验证层：

| 分类 | 执行的层 | 可跳过的层 |
|------|---------|-----------|
| **UI** | 契约 + 红线 + 静态 + 运行时（交互验证） | — |
| **样式** | 红线 + 静态（视觉循环在阶段 5 已完成） | 运行时 + AT |
| **数据** | 契约 + 红线 + 静态 + 运行时（migration + 数据完整性） | — |
| **API** | 契约 + 红线 + 静态 + 运行时（curl 调用验证） | UI 验证 |
| **逻辑** | 契约 + 红线 + 静态（单元/集成测试） | UI 验证 |
| **全栈** | 全部 5 层 | 无 |

**旅程连续性**（UI/数据/API/全栈类）：
- 上游衔接：从上一步传来的数据/状态是否正确
- 下游传递：这一步的输出下一步是否能消费
- 旁路影响：分支路径是否都正常

**验证失败** → 回到阶段 5 修复，不要跳过。
