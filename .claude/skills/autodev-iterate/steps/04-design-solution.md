# 阶段 4：设计方案（plan mode）

**所有修改都必须先展示方案再实现。** 根据复杂度选择方案详细程度：

- **简单修复**（1-2 文件，改动明确）→ 简要方案（向用户展示：改哪个文件、改什么、为什么），确认后实现
- **涉及 UI 交互**（任何用户可见的改动）→ **必须** plan mode，方案须包含交互设计
- **复杂修复**（涉及架构变更）→ plan mode + 用户讨论

**方案必须包含**：
1. 要改的文件和具体改动
2. 为什么这样改（根因 → 方案的逻辑链）
3. **交互设计**（如涉及 UI）：具体的手势、状态变化、视觉反馈
4. **功能迁移形态**（如涉及"把功能从 A 移到 B"）：
   - 目标页面当前有多少信息？加入后会不会过载？
   - 评估三种形态：**内联**（直接放入现有列表） vs **子页面**（点击跳转） vs **展开面板**（点击展开详情）
   - 决策标准：<=3 项设置 → 内联；4-8 项 → 展开面板；>8 项或需要预览效果 → 子页面
   - 新增内容的视觉风格必须与目标页面现有元素一致
5. 验证方式

## 阶段 4.5：设计文档预更新（方案确认后、写代码前）

**方案确认后，必须先更新设计文档再开始写代码。** 这是防止"做完忘了同步"的核心机制。

检查以下每一项，按需更新：

| 条件 | 更新哪个文档 | 更新什么 |
|------|------------|---------|
| 方案改变了架构/技术选型 | `*-design.md` | 追加新章节描述变更 |
| 方案改变了 API 接口/数据模型 | `*-api.md` | 追加新端点或字段 |
| 方案引入了新依赖/新 prompt | `*-rules.md` | 补充依赖版本和约定 |
| 方案新增了功能 | `*-ideation.md` | 补充功能条目 |
| 方案存在设计漂移（实现偏离了原设计） | `*-ideation.md` 设计漂移表 | 记录漂移及保留原因 |

**违反检测**：如果阶段 5 开始 Edit/Write 代码文件但阶段 4.5 没有执行（没有任何文档被更新），必须停下来先补文档。

**输出**：列出更新了哪些文档的哪些章节，然后输出 `--- 阶段 4.5 完成 ---`。如果方案不涉及任何文档变更（纯 bug 修复、纯样式调整），输出"无文档需要更新"即可。

- `--- 阶段 4 完成 ---`（包含 4.5）
