# Phase 0：多条迭代编排

当用户一次性提出 **多条修改**（>=2 条）时，**不要合并处理**，必须走编排流程：

```
用户输入 N 条修改
        ↓
Phase 0: 编排计划
  - 列出所有修改项，编号
  - 评估每条的独立性和依赖关系
  - 排定执行顺序（有依赖的排后面）
  - 向用户确认顺序
        ↓
  ┌─ 第 1 条 ─→ 完整 iterate 流程（阶段 1-7）─→ 告知用户完成
  │
  ├─ 第 2 条 ─→ 完整 iterate 流程（阶段 1-7）─→ 告知用户完成
  │
  └─ 第 N 条 ─→ 完整 iterate 流程（阶段 1-7）─→ 告知用户完成
        ↓
    全部完成，总结
```

**Phase 0 具体步骤：**

1. **拆分**：将用户输入拆成独立的修改项，每条一句话描述
2. **排序**：
   - 依赖关系：如果 B 依赖 A 的改动结果，A 排前面
   - 风险排序：高风险/大范围改动排前面（趁代码状态干净时做）
   - 无依赖的按用户列出的顺序
3. **展示计划并确认**：
   ```
   迭代计划（共 N 条）：
   1. [简述] — 涉及: {预估文件范围}
   2. [简述] — 涉及: {预估文件范围}
   ...
   执行顺序：1 → 2 → ... → N
   是否按此顺序执行？
   ```
4. **逐条执行**：每条走完整的阶段 1-7，完成后向用户汇报，再开始下一条
5. **阶段 7 文档同步**可以在每条完成后做，也可以在全部完成后统一做（如果多条改动涉及同一文档，统一做更高效）

**为什么逐条执行：**
- 每条独立验证，出问题不会波及其他修改
- 用户可以在任意两条之间介入、调整方向、甚至取消后续
- 如果某条验证失败需要返工，不影响已完成的
- 每条的 git diff 更清晰，方便 review

**例外**：如果多条修改本质上是同一个根因的不同表现（如"按钮没反应"和"提交后无提示"都是同一个事件绑定 bug），可以合并为一条处理。但必须向用户说明合并原因。
