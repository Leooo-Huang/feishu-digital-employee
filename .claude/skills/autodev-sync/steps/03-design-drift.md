# 步骤 2.5：设计漂移检测

> **执行时机**：当由 autodev-iterate 阶段 7b 调用时，此步骤在阶段 7a（设计漂移修正）完成后执行，因此检测到的漂移应该是修正后的残余漂移。独立使用时，漂移检测结果可用于指导后续修正。

除了检查功能完成状态（步骤 2），还要检查**设计决策是否仍与代码一致**。设计文档中的技术选型、API 定义、数据模型可能在开发过程中被修改但文档未同步更新，导致文档成为过时的"谎言"。

**检测项：**

| 维度 | 文档来源 | 代码来源 | 检测方法 |
|------|---------|---------|---------|
| 技术选型 | design.md 技术选型表中的依赖和版本 | `package.json` / `pyproject.toml` 实际依赖 | 对比依赖名称和版本号 |
| API 端点 | api.md 中的端点路径和 HTTP 方法 | 代码中的路由定义（`app/api/`、路由注册文件） | Grep 路由注册模式 |
| 数据模型 | api.md / design.md 中的实体和字段定义 | schema/model 定义文件（Prisma schema、TypeORM entity 等） | 对比实体名和字段名 |
| 页面路由 | ui.md 中的页面清单 | `app/` 或 `pages/` 目录下的路由文件 | Glob 匹配路由文件 |
| 核心依赖 | design.md 中指定的库/框架 | 实际 import 的库 | Grep import 语句 |

**漂移分级：**

| 级别 | 定义 | 处理方式 |
|------|------|---------|
| `INFO` | 代码中有但文档没提到的新增项（如新依赖、新端点、新页面） | 在同步报告中标注，建议补充到对应文档 |
| `WARN` | 文档说用 A 但代码实际用了 B（技术选型漂移、接口签名变化） | 在同步报告中高亮，要求用户确认：是更新文档还是修正代码？ |
| `ERROR` | 文档定义的核心接口/数据模型与代码完全不匹配（如文档定义了 5 个字段但代码只有 2 个同名字段） | 阻塞状态同步，要求先解决漂移再继续 |

**执行规则：**
- 只检测已有设计文档覆盖的维度。如果项目没有 api.md，跳过 API 端点检测
- 版本号比较时，只关注主版本号和次版本号差异（patch 版本差异标为 INFO）
- 如果 `*-index.md`（项目全景索引）和 `*-rules.md`（编码规则手册）存在，也检测它们与源设计文档的一致性

详细检测清单见 `checklists/drift-detection.md`。
