# 步骤 2：扫描代码实现

读取代码库，判断每个追踪项的实现状态：

**判断方法：**

| 追踪项类型 | 怎么判断已实现 |
|-----------|--------------|
| 功能 | 搜索相关组件/模块是否存在，关键逻辑是否实现 |
| 页面 | 搜索对应的页面文件/路由是否存在 |
| API 端点 | 搜索对应的路由定义、controller/handler 是否存在 |
| 数据模型 | 搜索 schema/model 定义是否存在 |
| 计划任务 | 读取 git log，查找相关提交 |

**辅助信号：**
- `git log --oneline` — 查找与功能相关的提交信息
- 文件存在性 — 对应文件是否已创建
- 代码搜索 — 关键函数/类/组件是否存在
- 测试覆盖 — 对应测试文件是否存在且通过

**桩代码检测（必须在判断状态之前执行）：**

对每个追踪项的关键代码文件做桩代码扫描。详细清单见 `checklists/stub-detection.md`。

**规则**：存在桩代码信号的功能，必须在备注中注明具体的桩代码位置。如果核心数据管道完全断裂（如数据源硬编码为空），应标为 `[ ]` 而非 `[~]`。

**实现程度分级：**

| 状态 | 标记 | 判断标准 |
|------|------|---------|
| 已完成 | `[x]` | 代码存在 + 核心逻辑可运行 + 测试存在（或明显完整） + 无桩代码信号 |
| 部分完成 | `[~]` | 代码存在且核心逻辑可运行，但功能不完整或缺少测试。注意：核心数据管道断裂（数据源为空/桩/未接通）应标 `[ ]` 而非 `[~]` |
| 未开始 | `[ ]` | 无相关代码，或代码存在但核心数据管道完全断裂（桩代码） |
| 已跳过 | `[-]` | 明确标记为不做 |
