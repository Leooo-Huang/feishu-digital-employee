# 步骤 4：阶段 2 — 产品规格

**目标**：先做 MECE 能力分解，再调用 brainstorming 生成产品规格。

## 步骤 2a：MECE 能力分解（不可跳过）

<IMPORTANT>
这一步是阶段 2 的前置步骤，不可跳过。目的是防止后续架构选型和技术决策在"维度混乱"的功能定义上做出错误判断，导致选错技术、组件重叠或能力遗漏。
</IMPORTANT>

读取 `docs/plans/*-ideation.md`，执行以下分析：

### 1. 提取核心价值链
从 ideation 文档的「本质任务分析」章节读取核心价值链（步骤 1.5 Q2 的产出）。

### 2. 按单一维度做能力分解
选择**一个**维度（推荐按"系统能力"），把系统需要的能力拆成互不重叠的条目。

**分解规则：**
- **单一维度**：所有条目必须在同一个分类维度上
- **互斥**：每个能力条目不能和其他条目有交叉
- **穷尽**：核心价值链的每个环节都必须被至少一个能力条目覆盖

**产出格式：**
```
能力清单（维度：系统能力）
R1: [必需能力名称] — [一句话描述]
R2: [必需能力名称] — [一句话描述]
...
O1: [可选能力名称] — [一句话描述]
O2: [可选能力名称] — [一句话描述]
...
```

### 3. 标记必需 vs 可选
- **必需（R）**：没有它系统就不完整（对应核心价值链的必要环节）
- **可选（O）**：有了更好，但 MVP 可以没有

### 4. 交叉验证

| 检查项 | 通过条件 |
|--------|---------|
| 核心价值链的每个环节是否都有对应的能力条目？ | 全覆盖 |
| ideation 的失败模式（Q3）是否都有对应的防护能力？ | 全覆盖 |
| 能力条目之间是否有重叠？ | 无重叠 |
| 是否混用了不同维度？ | 单一维度 |

如果检查不通过，调整分解直到通过。

## 步骤 2a.5：开源方案扫描（不可跳过）

<IMPORTANT>
这一步是为了避免重复造轮子。每个必需能力（R 编号）在决定自研之前，必须先扫描近期活跃的开源方案，由 brainstorming 决定"复用 vs 自研"。
</IMPORTANT>

### 1. 对每个 R 能力做开源调研

对能力清单中的每个 `R1, R2, ...`，执行：

**a) 提取能力关键词**：从能力描述提炼 1-3 个英文搜索词（如"实时协作" → `realtime collaboration`、`CRDT`）。

**b) 调用 `last30days` 技能**：
- 调用方式：`Skill(skill="last30days", args="{能力关键词} open source framework library")`
- 目的：获取 Reddit/X/YouTube/web 上近 30 天被讨论的热门方案
- 无法调用时（离线/rate limit）：fallback 到 WebSearch

**c) WebSearch 补充历史沉淀方案**：
- 搜索：`"top {能力关键词} open source {当前年份}"`、`"best {能力关键词} library {当前年份}"`
- 目的：补充 last30days 未覆盖的成熟老牌方案

**d) 对每个候选方案记录**：

| 字段 | 说明 |
|------|------|
| 名称 | 库/框架名 |
| 仓库 | GitHub URL |
| Stars | 当前 star 数 |
| 最近 commit | `git log` 最近一次提交日期（通过 WebFetch GitHub 仓库主页） |
| License | MIT / Apache-2.0 / AGPL / 商业 等 |
| 近期热度 | last30days 命中 / 未命中 |
| 适配度评估 | 1-5 分：功能覆盖本 R 能力的程度（5 = 完全覆盖，1 = 仅覆盖一小部分） |
| 集成成本 | low / medium / high（依据依赖数、配置复杂度、是否需要改造） |

### 2. 产出开源扫描清单

写入 `docs/plans/YYYY-MM-DD-{slug}-oss-scan.md`，按 R 编号分章节：

```markdown
# 开源方案扫描清单

## R1: {能力名}

关键词：`{搜索词}`

### 候选方案

| 名称 | Stars | 最近 commit | License | 热度 | 适配度 | 集成成本 | 链接 |
|------|-------|-----------|---------|------|--------|----------|------|
| Lib-A | 12.3k | 2026-04-20 | MIT | 🔥 | 5/5 | low | github.com/... |
| Lib-B | 5.1k  | 2026-03-15 | Apache-2.0 | — | 4/5 | medium | github.com/... |

### 推荐

**优先选用**：Lib-A（适配度最高 + 许可证友好 + 近期活跃）

### 自研保留理由（如适用）

仅当所有候选都不满足时填写，必须列出具体差距。

## R2: ...
```

### 3. 淘汰规则（硬性）

下列候选不进入 brainstorming 的可选池：
- License 与项目不兼容（商业场景下的 AGPL、GPL）
- 最近 commit 超过 12 个月（abandonware 风险）
- Stars < 100 且非新项目（< 6 个月）
- 适配度 ≤ 2（覆盖面太窄）

### 4. 交叉验证

| 检查项 | 通过条件 |
|--------|---------|
| 每个 R 能力是否至少扫描了 3 个候选？ | 至少 3 个（除非该能力极小众） |
| 每个 R 能力是否有"推荐"结论？ | 全部有 |
| 自研的 R 能力是否填写了差距说明？ | 全部填写 |

## 步骤 2b：调用 brainstorming

调用 `superpowers:brainstorming` 技能。

**衔接：** 读取 `docs/plans/*-ideation.md` 中的 MVP 部分，分别提取**平台层功能**（P 编号）和**业务场景层功能**（W 编号）及其用户故事、依赖关系，**同时传递步骤 2a 的能力清单 + 步骤 2a.5 的开源扫描清单**，作为 brainstorming 的输入上下文。告诉 brainstorming：

> "这是经过调研确定的分层功能清单——平台层是所有场景共享的基础能力，场景层是端到端业务流程。以下是 MECE 能力分解清单，请确保架构设计覆盖所有必需能力（R 编号），每个必需能力在架构中有明确的主责组件。
>
> 此外，附上开源方案扫描清单 `*-oss-scan.md`。**默认立场：优先复用清单中"推荐"的开源方案，不自写**。只有当开源方案都不满足适配度、license、活跃度要求时，才能选择自研，且必须在设计文档中显式写出"自研理由"章节（对比候选方案列出具体差距）。"

brainstorming 产出的设计文档应包含：
- **能力-组件映射表**：每个必需能力（R 编号）由哪个组件主责实现，**并标注"复用 Lib-X / 自研"**
- **复用决策表**：每个 R 能力的"复用 vs 自研"决策及依据
- 功能详细规格（展开 ideation 的 MVP 功能）
- 架构决策和技术选型
- 数据模型概要
- 核心组件划分
- **部署策略**（平台选择、环境配置、CI/CD 方案）

## 技术鲜度验证（红线 4 检查点）

<IMPORTANT>
brainstorming 做出技术选型决策后、写入设计文档之前，必须对每个核心依赖执行 WebSearch 验证：

1. **框架/运行时**：搜索 `"{框架名} latest stable version {当前年份}"` → 确认选用的是最新稳定版
2. **核心库**（UI 库、ORM、状态管理等）：搜索 `"{库名} latest version"` → 确认版本号
3. **API/SDK**：搜索 `"{服务名} API latest version deprecated"` → 确认 API 未被废弃
4. **替代品检查**：搜索 `"{库名} vs alternatives {当前年份}"` → 确认没有更好的主流替代品

验证结果必须写入设计文档的技术选型章节，格式：
```
| 依赖 | 选用版本 | 最新稳定版 | 查证日期 | 选型理由 |
```

如果发现训练数据中的记忆版本与 WebSearch 结果不一致，**以 WebSearch 结果为准**。
</IMPORTANT>

## 设计文档验证（brainstorming 产出后）

在标记阶段 2 完成之前，检查：
- 能力清单中每个 R 编号的能力，在设计文档中是否有对应的组件/模块？
- 有没有组件覆盖了多个能力但没有说明主次关系？
- 如果某个能力被多个组件覆盖，是否说明了它们是主次/互补/不同子场景的关系？

如果检查不通过，要求补充或修正设计文档后再继续。

- **输入**：`*-ideation.md` + 能力清单 + `*-oss-scan.md`
- **产出**：
  - `docs/plans/YYYY-MM-DD-{slug}-oss-scan.md`（开源扫描清单，2a.5 产出）
  - `docs/plans/YYYY-MM-DD-{slug}-design.md`（含能力-组件映射表 + 复用决策表）
